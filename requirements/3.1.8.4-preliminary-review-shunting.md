# 3.1.8.4 Preliminary Review Submission & Smart Shunting of Performance Grades — Frontend Requirements

**Source:** F-PERF-004 — Section 3.1.8 Performance Appraisal & Scoring  
**Route:** `/pms/appraisal/preliminary` (row actions + batch panel)  
**Workflow:** Extends [3.1.8.3](./3.1.8.3-preliminary-review-list.md) on same route → routes A/B to [3.1.8.7](./3.1.8.7-executive-final-review.md); C/D to [3.1.8.5](./3.1.8.5-hr-rectification.md) → [3.1.8.6](./3.1.8.6-secondary-confirmation.md)

---

## Functional Summary

Auditor confirms preliminary review results; system routes employees by grade — high performers to CFO final review, mid/low performers to HR assistance queue.

**Input:** Employee ID, performance grade confirmed at preliminary review.  
**Prerequisites:** Employee status is "Pending Preliminary Review".

---

## Phase 1: Deep Requirement Analysis

### User Personas

| Persona | Goals | Pain Points |
|---------|-------|-------------|
| **Auditor** | Confirm grades and batch-submit with clear routing visibility | Manual email handoffs to HR vs CFO |
| **HR Specialist** | Receive C/D employees promptly in HR queue | Delayed notification of low performers |
| **CFO / Executive** | See only actionable A/B items in final queue | Clutter from HR-path employees |

### Core Capabilities (FR-001…)

| ID | Capability | Priority |
|----|------------|----------|
| FR-001 | Per-row grade confirmation (dropdown A/B/C/D, default = auto grade) | Must |
| FR-002 | Optional review opinion per employee | Should |
| FR-003 | Shunting: A/B → pending final review (CFO); C/D → pending HR processing | Must |
| FR-004 | Visual routing diagram explaining shunt rules | Must |
| FR-005 | Batch multi-select submit with destination summary | Must |
| FR-006 | Immutable shunting audit log | Must |
| FR-007 | Update nav badge counts on `/pms/appraisal/hr` and `/pms/appraisal/final` (mock) | Should |
| FR-008 | Block confirm on already-shunted rows | Must |

### Edge Cases & Alternative Paths

- **Confirmed grade differs from auto grade:** Allowed; audit log records both.
- **Batch mix of A/B and C/D:** Summary shows split counts before confirm.
- **Single row confirm without opinion:** Allowed.
- **Batch with some already shunted:** Skip shunted; warn in summary.
- **Network failure mid-batch:** Show partial success; failed rows remain selectable.
- **All selected same destination:** Simplified summary copy.
- **CFO policy — C/D on final list read-only:** Documented in routing diagram; enforced in [3.1.8.7](./3.1.8.7-executive-final-review.md).

### Shunting Rules

| Confirmed grade | Next queue | CFO visibility |
|-----------------|------------|----------------|
| A, B | Pending final review (CFO) | Visible, actionable |
| C, D | Pending HR processing | Visible on CFO list as read-only / hidden per policy (mock: "HR queue — not actionable") |

## Phase 2: Screen-by-Screen Functional Specifications

### Screen: Preliminary Review — Shunting Actions (embedded on `/pms/appraisal/preliminary`)

* **Objective:** On the list from [3.1.8.3](./3.1.8.3-preliminary-review-list.md), enable per-row and batch grade confirmation with smart routing.
* **UI Elements & Field-Level Specs:**
  * **Extended table columns** (pending rows only): Confirmed grade (inline dropdown A/B/C/D), Review opinion (expandable textarea, optional), **Confirm** row button
  * Shunted rows: read-only confirmed grade; status badge; no confirm button
  * **Batch panel** (sticky bottom when ≥1 row selected): Selected count, breakdown "→ CFO: N | → HR: M", checkbox "Use auto grade for all", **Submit selected** primary
  * **Shunting rules** info card: visual routing diagram (A/B → CFO, C/D → HR) with links to [3.1.8.7](./3.1.8.7-executive-final-review.md) and [3.1.8.5](./3.1.8.5-hr-rectification.md)
  * Tab or side panel toggle: **Audit log**
* **Functional Logic:**
  * Confirm single row → mock API → toast with destination label → update row status
  * Increment mock badge counts on HR and Final nav items
  * Create operational log entry per shunt

### Panel: Shunting Audit Log (tab on preliminary page)

* **Objective:** Immutable trail of all shunting decisions.
* **UI Elements & Field-Level Specs:**
  * Table: Timestamp, Employee, Confirmed grade, From status, To status, Auditor, Opinion snippet (truncated, expand on hover)
  * Filter: Date range, Auditor, Destination (CFO / HR)
  * Read-only; no delete
* **Functional Logic:** Append-only mock store; paginated newest-first

### Modal: Batch Submit Confirmation

* **Objective:** Review batch impact before committing shunts.
* **UI Elements & Field-Level Specs:**
  * Summary table: Total selected, → CFO count, → HR count
  * Grade override summary if bulk auto-grade applied
  * **Confirm batch** / **Cancel**
  * Progress bar during submit (mock)
* **Functional Logic:**
  * On confirm: sequential mock updates; close with success toast + summary
  * Refresh table and audit log

### Modal: Single Confirm Success Toast / Inline

* **Objective:** Immediate feedback on routing destination.
* **UI Elements:** "Routed to CFO final review" or "Routed to HR assistance queue" with optional **View queue** link
* **Functional Logic:** Link navigates to `/pms/appraisal/final` or `/pms/appraisal/hr`
