# 3.1.1.9 Operation Log Query and Traceability — Frontend Requirements

**Source:** F-BASE-009 — Section 3.1.1 System Basic Management  
**Route:** `/pms/admin/logs`

---

## Phase 1: Deep Requirement Analysis

### User Personas

| Persona | Goals |
|---------|-------|
| **Auditor / Compliance Officer (primary)** | Query critical operations by operator, time, business type, and IP; inspect before/after data for investigations. |
| **Supreme System Administrator** | Export logs for external audit and trigger archive workflow (mock). |
| **Security Analyst** | Trace suspicious login IPs and correlate actions across business types. |
| **Operations Manager (read-only)** | Review changes affecting their domain (org, user, KPI) without mutating logs. |

### Core Capabilities

1. **FR-001 — Multi-condition filter:** Operator autocomplete, date range, business type multi-select, login IP text — aligned with `OperationLogSchema` in `src/models/pms/configuration.ts`.
2. **FR-002 — Paginated log table:** Sortable columns — Time, Operator, Business type, Target, Action, IP, Detail link.
3. **FR-003 — Detail inspection:** **Sheet** drawer with metadata and before/after JSON diff viewer (mock highlighted fields).
4. **FR-004 — Immutability:** No edit or delete controls anywhere in UI.
5. **FR-005 — Export mock:** **Button** triggers mock CSV download of current filter result set.
6. **FR-006 — Archive mock:** **Button** opens confirmation **Dialog** for archive action (no actual deletion).
7. **FR-007 — Permission gate:** System audit or supreme management permission required.
8. **FR-008 — Async states:** Loading, empty (no results), error, success for search and detail fetch.

### Edge Cases & Alternative Paths

- No filters applied → default last 7 days date range pre-filled.
- Search returns 0 rows → empty **Table** state "No logs match filters" with clear-filters link.
- Detail row has `BeforeData = null` (create action) → show "No prior state" in diff panel.
- `AfterData = null` (delete action) → show "No new state."
- Very large JSON payload → collapsible sections in diff viewer; copy JSON button.
- Export with 10k+ rows (mock) → **Toast** "Export started (mock)" without blocking UI.
- User opens detail **Sheet** then changes filters → close sheet or warn "Detail may be stale."
- Invalid date range (end before start) → inline validation on filter bar.
- Double-click Search → disabled during fetch.

## Phase 2: Screen-by-Screen Functional Specifications

### Screen Name: Operation Log Query (`/pms/admin/logs`)

* **Objective:** Search, browse, and export immutable operation logs.

* **UI Elements & Field-Level Specs:**
  * **Page header:** "Operation Logs"; audit permission badge.
  * **Filter bar (card layout):**
    * **Operator account (`Combobox` autocomplete):** Optional; searches mock operator list.
    * **Operation time range (`DatePicker` range):** Required default last 7 days; max range mock 1 year.
    * **Business type (`Multi-select`):** Options: org, user, role, dictionary, parameter, KPI, project, etc.
    * **Login IP (`Input`):** Optional; partial match; max 45 chars.
    * **Search (`Button`):** Primary; triggers query.
    * **Reset filters (`Button` ghost):** Clears to defaults.
  * **Toolbar above table:**
    * **Export (`Button` outline):** Mock CSV of current result set.
    * **Archive (`Button` outline):** Opens Archive Confirmation Dialog.
    * **Result count label:** "{N} records."
  * **Log `Table`:** Columns OperatedAt, OperatorAccount, BusinessType, TargetLabel, Action, LoginIp, Detail (link **Button** ghost "View").
  * **Pagination:** Page size 25/50/100; server-side mock.
  * **Loading:** **Table** skeleton.
  * **Empty:** Illustration + adjust filters hint.
  * **Error:** **Alert** + retry.

* **Functional Logic:**
  * Data from `src/mocks/pms/operation-logs.ts`.
  * Search builds query params; debounce not applied to Search button (explicit submit).
  * Row Detail click → open Log Detail **Sheet**.
  * Export uses current filters; **Toast** on click.
  * No delete/edit actions rendered.

### Screen Name: Log Detail Sheet (`Sheet` on `/pms/admin/logs`)

* **Objective:** Display full operation metadata and before/after data comparison.

* **UI Elements & Field-Level Specs:**
  * **Sheet (shadcn `Sheet`):** `side="right"`; `className="sm:max-w-xl"` or wider on desktop.
  * **Header:** Action + TargetLabel; OperatedAt subtitle.
  * **Metadata section:** OperatorAccount, BusinessType, LoginIp, Log Id — description list.
  * **Before / After diff viewer:** Two-column or unified diff; changed fields highlighted (mock green/red); monospace `ScrollArea`.
  * **Copy JSON (`Button` ghost):** Copies `AfterData` or full payload.
  * **Close (`SheetClose` or footer **Button`).**
  * **No edit/delete controls** — read-only by design.

* **Functional Logic:**
  * Lazy-load full log by `Id` on open if list row is summary only.
  * Handle null Before/After gracefully with placeholder text.
  * ESC and overlay close dismiss sheet.

### Screen Name: Archive Confirmation Dialog (modal on `/pms/admin/logs`)

* **Objective:** Mock acknowledgment of log archive policy without deleting UI data.

* **UI Elements & Field-Level Specs:**
  * **Dialog:** Title "Archive logs older than retention period?"; explains mock archive to cold storage.
  * **Retention note (`Alert`):** Informational only.
  * **Cancel / Confirm Archive (`Button`).**

* **Functional Logic:**
  * Confirm → **Toast** "Archive job queued (mock)"; no rows removed from **Table** (immutability demo).
