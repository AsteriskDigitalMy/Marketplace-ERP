# 3.1.4.3 Multi-Level Review of Filled Data — Frontend Requirements

**Source:** F-DATA-003 — Section 3.1.4 Data Collection and Filling  
**Route:** `/pms/data-collection/reviews`, `/pms/data-collection/reviews/:recordId`  
**UI:** shadcn/ui — `DataTable`, `Tabs`, `Card`, `Badge`, `Button`, `RadioGroup`, `Textarea`, `Form`, `Sheet`, `ScrollArea`, `Avatar`, `Separator`, `Toast`, `Skeleton`

---

## Functional Summary

Team leaders and department heads review submitted data level-by-level. Pass → locked into database for KPI calc; reject → return to filler with reason. Supports in-workflow messaging (mock).

**Prerequisites:** Data review permission; status `Pending Review`.

---

## Phase 1: Deep Requirement Analysis

### User Personas

| Persona | Role | Goals | Pain points |
|---------|------|-------|-------------|
| **Team Leader** | First-level reviewer | Quickly approve routine submissions; reject with clear reasons | Queue mixed with dept-level items |
| **Department Head** | Second-level reviewer | Ensure data quality before KPI calculation | No audit trail of prior level decisions |
| **Data Filler** | Submitter (read-only here) | Understand rejection reasons via messages | Slow back-and-forth outside system |
| **PMS Admin** | Oversight | Monitor review backlog by level | No visibility into bottlenecks |

### Core Capabilities

| ID | Capability | Description |
|----|------------|-------------|
| FR-001 | Multi-level review queue | Filter by team / department level, period, submitter, status |
| FR-002 | Review detail with read-only data | Display filled values + validation summary against thresholds |
| FR-003 | Pass action | Status → `approved`; "Locked for calculation" badge; eligible for [3.1.5.1](./3.1.5.1-scheduled-batch-calculation.md) |
| FR-004 | Reject action | Opinion required; status → `rejected`; notify filler (mock toast + message) |
| FR-005 | In-workflow messaging | Thread panel for Q&A between reviewer, filler, and other parties |
| FR-006 | Level routing | Team pass may advance to department queue (mock state machine) |
| FR-007 | Audit visibility | Show submission timestamp, prior level reviewer, and opinion history |

### Edge Cases & Alternative Paths

- **Reviewer lacks permission for level:** Record hidden from queue or row disabled with tooltip.
- **Pass at final level:** Data locked; no further edits without admin unlock mock.
- **Reject without opinion:** Form validation blocks submit.
- **Filler replies in message thread:** Reviewer receives mock notification badge.
- **Duplicate review attempt:** Optimistic lock — toast "Already processed" if status changed.
- **Empty queue:** Empty state "No records pending review."
- **Approved record opened:** Read-only detail; Pass/Reject hidden.
- **Concurrent reviewers:** Second reviewer sees updated status on refresh.

## Phase 2: Screen-by-Screen Functional Specifications

### Screen Name: Review Queue (`/pms/data-collection/reviews`)

* **Objective:** Reviewer finds and opens pending filling records for their level.
* **UI Elements & Field-Level Specs:**
  * Page header: "Data Review Queue".
  * Filter bar: `Select` review level (team / department), `Select` period, `Input` submitter search, `Select` status (default pending_review), `Button` "Reset filters".
  * `DataTable` columns: Record ID (mono), Indicator, Submitter, Period, Review level `Badge`, Submitted at, Status `Badge`, Actions (`Button` "Review").
  * Status badges: Pending (`secondary`), Approved (`default`), Rejected (`destructive`).
  * Pagination + sort by Submitted at desc.
* **Functional Logic:**
  * Fetch queue scoped to reviewer's permitted level(s).
  * Row click or Review → navigate to detail.
  * Tab counts mock: Pending (N), Completed (M).

### Screen Name: Review Detail (`/pms/data-collection/reviews/:recordId`)

* **Objective:** Reviewer inspects submission, converses via messages, and passes or rejects.
* **UI Elements & Field-Level Specs:**
  * Layout: two-column desktop (data left, actions right); stacked mobile.
  * Header: Indicator name, submitter, period, review level, submitted timestamp.
  * **Filled data `Card`:** read-only key-value grid or table of all field values.
  * **Validation summary `Card`:** each field with pass/fail icon vs KPI threshold (green check / red x).
  * **Decision `Card`** (hidden if already finalized):
    * `RadioGroup`: Pass / Reject.
    * Opinion `Textarea` — required when Reject selected; optional on Pass.
    * `Button` "Submit decision" primary; `Button` "Save draft opinion" outline (mock).
  * **Messaging panel `Card`:**
    * `ScrollArea` thread: avatar, author, timestamp, message text.
    * Footer: `Textarea` + `Button` "Send" for new message.
  * Approved state: `Badge` "Locked for KPI calculation" + lock icon.
* **Functional Logic:**
  * Pass → PATCH status approved → toast → if team level, mock advance to department queue OR lock if final.
  * Reject → validate opinion → PATCH rejected → mock notify filler with link to fill form.
  * Messages append to thread optimistically; rollback on error.

### Screen Name: Review History Tab (optional tab on queue page)

* **Objective:** Reviewer audits recently completed reviews.
* **UI Elements & Field-Level Specs:**
  * Same table columns minus Actions; Result column (Pass/Reject), Opinion truncated.
* **Functional Logic:**
  * Read-only; click row opens detail in read-only mode.

## Phase 3: Screen Transition & User Flow Mapping

| # | Trigger | Source | Destination | Conditional paths |
|---|---------|--------|-------------|-------------------|
| 1 | Nav "Data Review" | App shell | `/pms/data-collection/reviews` | No permission → 403 |
| 2 | Click "Review" | Queue row | `/pms/data-collection/reviews/:recordId` | — |
| 3 | Submit Pass | Review detail | Same page (approved state) | Team level → dept queue (mock) |
| 4 | Submit Reject | Review detail | Same page (rejected state) | Missing opinion → validation error |
| 5 | Reject notify | System mock | Filler message + fill form link | — |
| 6 | Send message | Messaging panel | Same page (thread update) | — |
| 7 | Filler resubmits | Fill form | Review queue (new pending) | — |
| 8 | Back to queue | Detail breadcrumb | `/pms/data-collection/reviews` | Unsaved opinion → confirm |
| 9 | Approved data consumed | Calculation job | [3.1.5.1](./3.1.5.1-scheduled-batch-calculation.md) | Out of scope for UI |
