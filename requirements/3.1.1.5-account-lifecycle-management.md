# 3.1.1.5 Account Full Lifecycle Management — Frontend Requirements

**Source:** F-BASE-005 — Section 3.1.1 System Basic Management  
**Route:** `/pms/admin/accounts/:id`

---

## Phase 1: Deep Requirement Analysis

### User Personas

| Persona | Goals |
|---------|-------|
| **System Administrator (primary)** | Manage full account lifecycle — activate, disable, reset password, edit HR info — with every action audited. |
| **HR Coordinator** | Keep employee department and position current without changing immutable employee ID. |
| **IT Security Officer** | Disable compromised accounts immediately and confirm sessions are cleared (mock). |
| **Auditor** | Review immutable operation log on the account detail page. |

### Core Capabilities

1. **FR-001 — Account detail view:** Display `UserAccount` fields from `src/models/pms/identity.ts` — name, employee ID, login account, status, department, roles, last login.
2. **FR-002 — Disable account:** Set `Status = disabled` with confirmation; mock session termination **Toast**.
3. **FR-003 — Activate account:** Set `Status = active` with confirmation; restore login capability.
4. **FR-004 — Password reset:** Generate mock new password; show in **Dialog** with copy button; mock notification sent **Toast**.
5. **FR-005 — HR info edit:** Update name, department, position, contact (mock); employee ID read-only.
6. **FR-006 — Operation log panel:** Immutable list of account actions with timestamp, operator, action, detail.
7. **FR-007 — Navigation to role binding:** Link/tab to `/pms/admin/accounts/:id/roles` ([3.1.1.7](./3.1.1.7-user-role-binding.md)).
8. **FR-008 — Async states:** Loading, not-found, error, and success for fetch and each lifecycle action.

### Edge Cases & Alternative Paths

- Account already disabled → **Disable** hidden; **Activate** shown.
- Disable self (current logged-in admin mock) → block with **Toast** "Cannot disable your own account."
- Reset password on disabled account → allow (mock) or warn — spec: allow with warning **Alert** in dialog.
- HR edit with no changes → Save disabled.
- Department picker selects disabled org → validation error.
- Account not found → 404 with link to `/pms/admin/accounts`.
- Double-click on Disable / Activate / Reset → action buttons disabled until complete.
- Operation log empty → "No operations recorded."

## Phase 2: Screen-by-Screen Functional Specifications

### Screen Name: Account Detail (`/pms/admin/accounts/:id`)

* **Objective:** Central hub for viewing account state, performing lifecycle actions, and reviewing audit trail.

* **UI Elements & Field-Level Specs:**
  * **Header card:** `EmployeeName` (h1); `EmployeeId`, `LoginAccount`; status **Badge** (Active = default, Disabled = secondary).
  * **Metadata row:** Department name, Position, `LastLoginAt` or "Never".
  * **Roles summary:** Chips of role names; link **Manage roles** → `/pms/admin/accounts/:id/roles`.
  * **Actions dropdown (`DropdownMenu`):** Disable, Activate, Reset password, Edit employee info — items shown/hidden by status.
  * **Tabs:** Overview (default), Operation Log.
  * **Overview tab — read-only fields:** All `UserAccountDto` scalars in description list layout.
  * **Operation Log tab — `Table`:** Columns At, Operator, Action, Detail; no edit/delete controls; paginated.
  * **Loading:** Header + tab skeleton.
  * **Not found:** Empty card + back link.

* **Functional Logic:**
  * Fetch `UserAccountDto` from mocks on mount.
  * Each action opens appropriate **Dialog** (see below) except Edit opens Edit Dialog directly.
  * After any mutation: refresh detail; prepend row to `OperationLog`; **Toast** outcome.

### Screen Name: Disable Account Dialog (modal on `/pms/admin/accounts/:id`)

* **Objective:** Confirm account deactivation and session clearance.

* **UI Elements & Field-Level Specs:**
  * **Dialog:** Title "Disable account?"; body names employee + login account.
  * **Warning (`Alert`):** "Active sessions will be terminated (mock)."
  * **Cancel / Confirm Disable (`Button` destructive).**

* **Functional Logic:**
  * Block if self-disable; **Toast** instead.
  * On success: status **Badge** → Disabled; **Toast** "Account disabled — session cleared (mock)."

### Screen Name: Activate Account Dialog (modal on `/pms/admin/accounts/:id`)

* **Objective:** Confirm re-enabling login for a disabled account.

* **UI Elements & Field-Level Specs:**
  * **Dialog:** Title "Activate account?"; standard confirm/cancel.

* **Functional Logic:**
  * `PATCH` mock → status Active → **Toast** success.

### Screen Name: Reset Password Dialog (modal on `/pms/admin/accounts/:id`)

* **Objective:** Display generated temporary password and confirm notification dispatch.

* **UI Elements & Field-Level Specs:**
  * **Dialog:** Title "Password reset"; read-only `Input` with generated password; **Copy** icon button.
  * **Note:** "Notification sent to employee (mock)."
  * **Close (`Button`):** Dismisses dialog.

* **Functional Logic:**
  * Triggered after confirm step (nested: Confirm → Generate → Show password).
  * Password masked by default with reveal toggle.

### Screen Name: Edit Employee Info Dialog (modal on `/pms/admin/accounts/:id`)

* **Objective:** Update mutable HR fields without changing employee ID.

* **UI Elements & Field-Level Specs:**
  * **Dialog:** Title "Edit employee information"; `max-w-md`.
  * **Employee name (`Input`):** Required; max 200.
  * **Employee ID (`Input`):** Read-only; disabled styling.
  * **Department (`Combobox` org tree):** Required; active units only.
  * **Position (`Input`):** Required; max 100.
  * **Contact info (`Input` optional):** Mock phone/email; max 100.
  * **Cancel / Save (`Button`).**

* **Functional Logic:**
  * Dirty tracking; Zod validate against `UserAccountSchema` partial pick.
  * Save → mock `PUT` → close → **Toast** "Employee info updated" → log entry.

## Phase 3: Screen Transition & User Flow Mapping

| # | Trigger Event | Source Screen | Destination Screen | Conditional Paths |
|---|---------------|---------------|--------------------|-------------------|
| 1 | Click account row View | Account List | Account Detail | — |
| 2 | Page load valid `:id` | — | Account Detail | Invalid → 404 |
| 3 | Actions → Disable | Account Detail | Disable Account Dialog | Self-disable → Toast block |
| 4 | Confirm disable | Disable Dialog | Account Detail | Status updated |
| 5 | Actions → Activate | Account Detail | Activate Account Dialog | Only when disabled |
| 6 | Actions → Reset password | Account Detail | Reset Password Dialog | Two-step confirm |
| 7 | Actions → Edit employee info | Account Detail | Edit Employee Info Dialog | — |
| 8 | Save HR edit | Edit Dialog | Account Detail | Toast + log update |
| 9 | Click **Manage roles** | Account Detail | User Role Binding (`/pms/admin/accounts/:id/roles`) | — |
| 10 | Switch to Operation Log tab | Account Detail | Account Detail (log tab) | — |
