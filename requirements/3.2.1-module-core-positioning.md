# 3.2.1 Module Core Positioning — Frontend Requirements

**Source:** Section 3.2.1 — Product Data Management (PDM/ePDM/GST)  
**Module role:** Foundational data source for ERP — products, materials, processes, BOM, and cost data for SCM, MES, and WMS.

---

## Overview

The PDM module is the **master data hub** of the ERP frontend. The UI must present a coherent navigation structure, shared entity models, and cross-module data relationships so users understand that product/material/process data flows from PDM into downstream modules.

## Module Dashboard (`/pdm`)

### Purpose

Landing page summarizing PDM health and surfacing cross-module data consistency messaging.

### UI components

- **KPI stat cards** (shadcn `Card`): active projects, designs in review, open sampling tasks, pending BOM approvals, pending change requests.
- **Pipeline funnel chart** (shadcn Chart): projects → design → sampling → finalization counts.
- **Recent activity feed**: latest version changes, approvals, and sync events (mocked).
- **Downstream sync status panel**: badges for SCM / MES / WMS sync state per finalized product (mocked `Synced` | `Pending` | `Failed`).

### Data (mock)

```ts
interface PdmDashboardDto {
  activeProjects: number
  designsInReview: number
  openSamplingTasks: number
  pendingBomApprovals: number
  pendingChangeRequests: number
  pipelineCounts: { stage: string; count: number }[]
  recentActivity: ActivityEntry[]
  syncStatus: { module: 'SCM' | 'MES' | 'WMS'; status: string; productCount: number }[]
}
```

### States

| State | Behavior |
|-------|----------|
| Loading | Skeleton cards + skeleton list |
| Empty | "No PDM data yet" with CTA to create first project |
| Error | Retry banner with human-readable message |

## Global PDM Behaviors

1. **Version badge** — display current version on all entity headers (`v3.2` format).
2. **Status chip** — consistent status colors: `Draft`, `In Review`, `Approved`, `Archived`, `Rejected`.
3. **Approval trail** — reusable `ApprovalTimeline` component showing workflow steps (mocked).
4. **Attachment panel** — reusable file list with upload/preview/download actions (mocked blob URLs).
5. **Audit log drawer** — `Sheet` showing who changed what and when (mocked `LogDto`-shaped entries).

## Non-Functional Frontend Requirements

| Requirement | Detail |
|-------------|--------|
| Responsive | Admin layout from 1024px+; key read-only views usable on tablet |
| Accessibility | WCAG 2.1 AA per project UI rules |
| i18n-ready | All labels in sentence case English; strings externalized for future locale |
| Performance | Paginate all lists; virtualize BOM tree for large hierarchies |

---

## Out of Scope (frontend)

- Actual sync to SCM/MES/WMS backends
- Real file storage — mock upload success and preview with placeholder URLs
- CAD file rendering — show file metadata + download; image preview for raster formats only
