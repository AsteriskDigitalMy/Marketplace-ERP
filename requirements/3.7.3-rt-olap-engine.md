# 3.7.3 Real-Time Performance (RT-OLAP) Engine — Frontend Requirements

**Source:** Section 3.7.3  
**Note:** Backend engine is out of scope; frontend must reflect refresh SLAs and perceived performance.

---

## Screens

### Engine status (`/bi/settings/engine`)

| Setting | UI |
|---------|-----|
| Executive/management refresh | 5–30 min slider |
| Site kanban refresh | 3–10 sec slider |
| ETL mode display | Real-time CDC + scheduled batch (read-only info) |

---

## Performance UX requirements

| Query type | Target (display in dev overlay) |
|------------|--------------------------------|
| Single-dimension drill | ≤ 1.5s |
| Multi-table aggregate | ≤ 3s |

**Loading patterns:**

- Skeleton charts matching final layout (not full-page spinner)
- Stale-while-revalidate: show last data with "Updating…" badge during refresh
- Query > 3s → progress bar with cancel (mock)

---

## Data freshness indicator

Global `BiRefreshIndicator` on every dashboard:

- Last refreshed timestamp
- Next refresh countdown
- Manual refresh button (respects role)

---

## Mock latency simulation

`useBiDashboard` accepts `simulateSlowQuery` for testing skeleton/timeout states.

---

## Acceptance Criteria

- [ ] Refresh interval configurable per tier in settings UI
- [ ] Skeleton loading matches chart layout
- [ ] Stale data indicator during re-fetch
- [ ] Performance target documented in dev overlay (optional toggle)
- [ ] Manual refresh on all dashboards
