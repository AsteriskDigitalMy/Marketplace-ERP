# 3.1.7.2 Exception Alert Closed-Loop Disposal Process — Frontend Requirements

**Source:** F-ALERT-002 — Section 3.1.7 Exception Alert & Management  
**Routes:** `/pms/alerts`, `/pms/alerts/:id/disposal`  
**UI:** shadcn/ui — `DataTable`, `Badge`, `Button`, `Card`, `Form`, `Textarea`, `Input`, `DatePicker`, `Stepper` (custom or `Tabs` variant), `ScrollArea`, `Separator`, `Alert`, `Toast`, `Skeleton`, `FileUpload` (mock)

---

## Functional Summary

After an exception alert fires, the business owner completes root-cause investigation, rectification, and optimization within a deadline — forming a full management closed loop.

**Input:** Alert record ID, cause, rectification measures, completion results.  
**Prerequisites:** Corresponding business exception alert has been triggered.

---

## Phase 1: Deep Requirement Analysis

### User Personas

| Persona | Role | Goals | Pain points |
|---------|------|-------|-------------|
| **Alert Owner** | Assigned responsible person | Document cause and rectification before deadline | Unclear steps; lost email threads |
| **Verifying Manager** | Supervisor | Approve or return work with opinion | No audit trail of prior attempts |
| **Ops Manager** | Inbox triage | Filter overdue and urgent items first | Alert pile without status |
| **Auditor** | Compliance | Read immutable disposal log | Post-hoc reconstruction |

### Core Capabilities

| ID | Capability | Description |
|----|------------|-------------|
| FR-001 | Alert inbox | Filterable list with level, deadline countdown, status, owner |
| FR-002 | Overdue escalation display | Visual escalated level + history when past deadline unprocessed |
| FR-003 | Three-step disposal workflow | Root cause → rectification plan → management verification |
| FR-004 | Status stepper | Triggered → Investigating → Rectifying → Pending verification → Closed |
| FR-005 | Draft saves | Owner can save partial step without advancing status |
| FR-006 | Verification pass/fail | Pass closes; fail returns to owner with comments |
| FR-007 | Disposal audit log | Chronological immutable entries on detail page |
| FR-008 | Deadline countdown | Badge with days/hours; destructive when overdue |
| FR-009 | Link to originating rule | Display rule summary from [3.1.7.1](./3.1.7.1-alert-rule-configuration.md) |

### Edge Cases & Alternative Paths

- **Owner not assigned:** Status open; manager mock-assign action (optional banner).
- **Fail verification:** Status returns to rectifying; deadline may reset (mock toast).
- **Pass verification:** Status closed; all steps read-only.
- **Submit step without required fields:** Inline validation blocks advance.
- **Overdue auto-escalate:** `EscalationHistory` entry + urgent badge without user action.
- **Closed alert opened:** Full read-only; log expanded by default.
- **Attachment mock only:** Store file names, no real upload.
- **Empty inbox:** Empty state "No open alerts."

## Phase 2: Screen-by-Screen Functional Specifications

### Screen Name: Alert Inbox (`/pms/alerts`)

* **Objective:** Users triage and open alert records for disposal.
* **UI Elements & Field-Level Specs:**
  * Page header: "Alerts", summary chips: Open (n), Overdue (n), Urgent (n).
  * Filter bar: `Select` status, `Select` level, `Select` type, `Checkbox` "Overdue only", search by ID/object `Input`.
  * `DataTable` columns: Alert ID (link), Type / object, Level `Badge`, Triggered at, Deadline (countdown `Badge`), Status `Badge`, Owner, Actions (`Button` "Dispose" / "View").
  * Countdown badges: neutral > 3 days, warning ≤ 3 days, destructive overdue.
  * Overdue rows: subtle `bg-destructive/5` + escalated level icon if applicable.
* **Functional Logic:**
  * Default sort: deadline asc for open items.
  * ID click → disposal page.
  * Filters persist in URL query mock.

### Screen Name: Disposal Workflow (`/pms/alerts/:id/disposal`)

* **Objective:** Owner and manager complete closed-loop disposal with audit trail.
* **UI Elements & Field-Level Specs:**
  * **Timeline header `Card`:**
    * Alert summary: type, object, level badge, rule name link.
    * `EscalationHistory` accordion if entries exist.
    * Horizontal **status stepper**: Triggered → Investigating → Rectifying → Pending verification → Closed (active step highlighted).
  * **Step 1 — Root cause** (owner, statuses open/investigating):
    * Problem cause — required `Textarea`.
    * Impact scope — optional `Textarea`.
    * `Button` "Save draft", `Button` "Submit" primary → advances to rectifying.
  * **Step 2 — Rectification plan** (owner, status rectifying):
    * Disposal / rectification measures — required `Textarea`.
    * Planned completion date — required `DatePicker`.
    * Attachments — mock file name list + `Button` "Add file".
    * Submit for confirmation → status `pending_verification`; mock notify manager banner.
  * **Step 3 — Management verification** (manager, status pending_verification):
    * Verification result — `RadioGroup` Pass / Fail.
    * Verification opinion — required `Textarea` on fail.
    * `Button` "Submit verification".
    * Pass → closed toast; Fail → return owner alert with comments.
  * **Disposal log panel `Card`:**
    * `ScrollArea` vertical timeline: timestamp, actor, action, detail.
    * Read-only; newest first or oldest first (config: oldest first for audit).
  * Closed state: all forms disabled; stepper shows Closed; success `Alert`.
* **Functional Logic:**
  * Load `AlertRecordDto` by id; skeleton header + step content.
  * Step visibility gated by `Status` and user role mock (owner vs manager).
  * Each submit appends to `DisposalLog` mock.
  * Overdue on load: show escalated badge if `IsOverdue` and not closed.

### Screen Name: Escalation Display (inline on disposal header)

* **Objective:** Surface auto-escalation when deadline missed.
* **UI Elements & Field-Level Specs:**
  * `Alert` variant warning: "Auto-escalated to Urgent at {timestamp}".
  * `Badge` showing current effective level.
* **Functional Logic:**
  * Render when `EscalationHistory.length > 0`.
