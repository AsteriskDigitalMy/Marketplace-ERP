# 3.1.6.3 Global Traffic Light Status Color Rule Configuration — Frontend Requirements

**Source:** F-DASH-003 — Section 3.1.6 KPI Visualization Monitoring Dashboard  
**Route:** `/pms/settings/traffic-lights`  
**Prerequisites:** User has system global parameter configuration permission (mock role gate)  
**UI:** shadcn/ui — `Card`, `Table`, `DataTable`, `Dialog`, `Form`, `Input`, `Select`, `Button`, `Badge`, `AlertDialog`, `Toast`, `Skeleton`, `Label`, `MultiSelect` (combobox pattern)

---

## Functional Summary

Uniformly customize color threshold standards for progress, achievement rates, and performance levels across the system. Saved rules sync to all dashboards and lists.

**Input:** Rule classification, numerical threshold ranges, display colors.

---

## Phase 1: Deep Requirement Analysis

### User Personas

| Persona | Role | Goals | Pain points |
|---------|------|-------|-------------|
| **System Admin** | Global config owner | Set consistent green/yellow/red bands once for all modules | Inconsistent colors between cockpit and lists |
| **PMS Admin** | KPI operator | Preview how thresholds affect display | Changing rules breaks existing dashboards |
| **End user (consumer)** | Read-only | Instantly recognize status severity | Arbitrary colors per screen |

### Core Capabilities

| ID | Capability | Description |
|----|------------|-------------|
| FR-001 | Rule category CRUD | Three categories: progress, delay, performance |
| FR-002 | Band range editor | Green, yellow, red each with min/max numeric bounds |
| FR-003 | Overlap validation | Inline errors if bands intersect |
| FR-004 | Metric scope assignment | Optional multi-select limiting rule to KPIs/modules |
| FR-005 | Live preview swatches | Sample badges update as admin types ranges |
| FR-006 | Propagation mock | Save updates in-memory store; consumers re-read on next render |
| FR-007 | Permission gate | Unauthorized users do not see settings page |
| FR-008 | Unsaved changes guard | `AlertDialog` on navigate away from dirty editor |
| FR-009 | Shared helper export | `getStatusColor(value, category)` for cockpit, lists, appraisal |

### Edge Cases & Alternative Paths

- **Overlapping ranges:** Save blocked with field-level errors.
- **Gap between bands:** Warning (non-blocking) "Values between X and Y have no color" optional.
- **Delete last rule in category:** Revert to system defaults mock on consumers.
- **Min > max:** Field validation error.
- **Negative delays or percentages:** Validate per category semantics.
- **Concurrent edit (mock):** Last save wins; toast if version stale.
- **Empty metric scope:** Rule applies globally within category.

## Phase 2: Screen-by-Screen Functional Specifications

### Screen Name: Traffic Light Rules List (`/pms/settings/traffic-lights`)

* **Objective:** Admin views all global color threshold rules and opens editor.
* **UI Elements & Field-Level Specs:**
  * Page header: "Traffic Light Rules", description text explaining system-wide effect.
  * `Button` "Add rule" primary.
  * Filter: `Select` category (progress / delay / performance / all).
  * `DataTable` columns: Category `Badge`, Metric scope (chips or "Global"), Green range, Yellow range, Red range, Last updated, Updated by, Actions (Edit, Delete).
  * Empty state: illustration + "Add rule" CTA.
  * Delete → `AlertDialog` confirm.
* **Functional Logic:**
  * Load `TrafficLightRuleDto[]` on mount.
  * Edit opens Rule Editor dialog with row data.
  * Delete removes from mock store; toast "Dashboards will update on next load."

### Screen Name: Rule Editor (Dialog on `/pms/settings/traffic-lights`)

* **Objective:** Create or edit one traffic-light rule with validated bands and preview.
* **UI Elements & Field-Level Specs:**
  * `Dialog` lg: title "Edit rule" / "Add rule".
  * `Form` fields:
    * Category — required `Select` (progress | delay | performance).
    * Metric scope — optional multi `Combobox` (KPI names, module tags).
    * **Green band:** Min `Input` number, Max `Input` number, color swatch `Badge` variant success.
    * **Yellow band:** Min, Max, swatch warning.
    * **Red band:** Min, Max, swatch destructive.
    * **Preview panel `Card`:** sample values (e.g. 95%, 80%, 50%) rendered through `getStatusColor` live.
  * Footer: Cancel, Save primary.
  * Overlap errors under affected band fields.
* **Functional Logic:**
  * On change, recompute preview badges.
  * Validate no overlap and min ≤ max per band.
  * Save → PATCH mock store → toast "Rules saved — dashboards will update" → close → refresh list.
  * Dirty form → confirm on cancel.

### Screen Name: Permission Denied

* **Objective:** Hide configuration from unauthorized roles.
* **UI Elements & Field-Level Specs:**
  * Full-page empty: lock icon, permission message, back to cockpit link.
* **Functional Logic:**
  * Route guard before data fetch.

### Consumer Integration (display only — no edit UI)

Components that consume rules:

- `/pms/cockpit` cards and drill-down ([3.1.6.1](./3.1.6.1-role-cockpit-auto-loading.md), [3.1.6.2](./3.1.6.2-cockpit-drill-down.md))
- Appraisal grade badges ([3.1.8.x](./3.1.8.1-appraisal-scheme-configuration.md))
- Alert severity color mapping where applicable

### Rule Categories Reference

| Category | Example metrics | Default bands |
|----------|-----------------|---------------|
| Progress achievement | KPI completion %, project progress | Green ≥ 90%, Yellow 70–89%, Red < 70% |
| Project delay | Days behind schedule | Green on time, Yellow 1–7 days, Red > 7 days |
| Performance level | Appraisal score | Green A/B, Yellow C, Red D |
