# 3.1.1.7 Fast Binding of User Roles — Frontend Requirements

**Source:** F-BASE-007 — Section 3.1.1 System Basic Management  
**Route:** `/pms/admin/accounts/:id/roles`

---

## Phase 1: Deep Requirement Analysis

### User Personas

| Persona | Goals |
|---------|-------|
| **System Administrator (primary)** | Bind one or many business roles to employee accounts quickly; preview effective permissions before saving. |
| **HR Coordinator** | Batch-assign roles after org restructuring without editing each account individually. |
| **Security Architect** | Ensure every account retains at least one role (`UserRoleBindingSchema` / `UserAccountSchema` min 1 role). |
| **End User (indirect)** | Sees updated menus/buttons immediately after binding save (mock refresh). |

### Core Capabilities

1. **FR-001 — Single-user role binding:** Checkbox list of `SystemRole` options on `/pms/admin/accounts/:id/roles`; multi-select; min 1 role.
2. **FR-002 — Effective permission preview:** Read-only merged menu/button tree reflecting selected roles before save.
3. **FR-003 — Save binding:** Persist `RoleIds` via `UserRoleBinding` model in `src/models/pms/identity.ts`; **Toast** "Permissions refreshed."
4. **FR-004 — Change log:** Append `UserRoleBindingDto` audit entry (ChangedAt, ChangedBy).
5. **FR-005 — Batch binding from account list:** Multi-select rows on `/pms/admin/accounts` → **Batch assign roles** dialog.
6. **FR-006 — Batch modes:** Add roles (union) vs Replace roles (overwrite) toggle in batch dialog.
7. **FR-007 — Batch progress:** Progress indicator and summary **Toast** after batch complete.
8. **FR-008 — Async states:** Loading, error, success for single and batch flows.

### Edge Cases & Alternative Paths

- User unchecks all roles → Save disabled; inline **Alert** "At least one role required."
- Batch assign with 0 accounts selected → toolbar action hidden/disabled.
- Batch replace mode with empty role selection → validation error.
- Selected user account disabled → allow role edit (mock) or read-only — spec: allow edit for future activation.
- Role deleted after page load → show stale role chip with warning; remove on save.
- User saves without changes → Save disabled.
- Double-click Save or batch Confirm → disabled until complete.
- Permission preview slow (mock) → skeleton tree in preview panel.

## Phase 2: Screen-by-Screen Functional Specifications

### Screen Name: User Role Binding (`/pms/admin/accounts/:id/roles`)

* **Objective:** Configure role membership for one account with permission preview.

* **UI Elements & Field-Level Specs:**
  * **Page header:** "Roles — {EmployeeName}"; breadcrumb: Accounts → Account Detail → Roles.
  * **Account summary strip:** Employee ID, login account, current role chips.
  * **Available roles panel (`ScrollArea`):** Checkbox per role showing Name + Remarks; search filter.
  * **Effective permissions preview (`ScrollArea`, right column or below on mobile):** Read-only tree — menus and button badges merged from selected roles.
  * **Preview legend:** Explains union of permissions across roles.
  * **Cancel (`Button` outline):** → `/pms/admin/accounts/:id`.
  * **Save (`Button`):** Disabled when invalid, pristine, or submitting.
  * **Recent changes `Table` (optional footer):** Last 5 `UserRoleBinding` entries for this user.

* **Functional Logic:**
  * Load account `RoleIds` + all roles from mocks.
  * On checkbox change: recompute preview tree (debounced 150ms).
  * Save: validate `UserRoleBindingSchema` → mock `PUT` → **Toast** → navigate back or stay.
  * Log binding change with actor/timestamp.

### Screen Name: Account List with Multi-Select (`/pms/admin/accounts`)

* **Objective:** Select multiple accounts for batch role assignment.

* **UI Elements & Field-Level Specs:**
  * **Table row checkbox column:** Select row; header checkbox select-all on page.
  * **Selection bar (sticky):** "{N} selected" + **Batch assign roles (`Button`)** + Clear selection.
  * **Batch assign disabled:** When N = 0.

* **Functional Logic:**
  * Selection state local; persists during pagination warning (mock: clear on page change with confirm).

### Screen Name: Batch Assign Roles Dialog (modal on `/pms/admin/accounts`)

* **Objective:** Apply role set to multiple accounts in add or replace mode.

* **UI Elements & Field-Level Specs:**
  * **Dialog:** Title "Batch assign roles"; subtitle "{N} accounts selected."
  * **Mode toggle (`RadioGroup`):** Add roles (keep existing, union) | Replace roles (overwrite all).
  * **Role checkboxes:** Same list as single-user binding; min 1 required.
  * **Summary line:** Dynamic text explaining mode effect.
  * **Cancel (`Button` outline).**
  * **Confirm (`Button`):** Shows progress spinner during mock batch job.

* **Functional Logic:**
  * Mock async per-account update with **Progress** bar.
  * Complete → **Toast** "{N} accounts updated"; summary Dialog optional: success/fail counts.
  * On partial failure (mock): **Toast** warning with fail count; failed accounts listed in **Table** inside dialog.
