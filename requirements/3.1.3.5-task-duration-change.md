# 3.1.3.5 Project Task Duration Change Application — Frontend Requirements

**Source:** F-PROJ-005 — Section 3.1.3 Project Management  
**Route:** `/pms/tasks/:id/duration-change`  
**Related:** [3.1.3.3](./3.1.3.3-sub-task-creation.md), [3.1.3.4](./3.1.3.4-project-progress-updates.md), [3.1.3.9](./3.1.3.9-gantt-chart.md) (Gantt refresh), [3.1.3.2](./3.1.3.2-project-initiation-approval.md) (leader role)

---

## Functional Summary

Executors request planned start/end adjustments; leader reviews. Approved changes update task duration with audit record.

**Prerequisites:** Task not completed.

---

## Phase 1: Deep Requirement Analysis

### User Personas

| Persona | Role | Goals | Pain Points |
|---------|------|-------|-------------|
| **Task Executor** | Submits change requests | Extend timeline with justified reason | No feedback on pending requests |
| **Project Leader** | Approves schedule changes | Maintain project end date integrity | Multiple pending requests per task |
| **Scheduler** | Views Gantt impact | See date shifts reflected visually | Manual Gantt edits not synced |

### Core Capabilities (FR-xxx)

| ID | Capability | Priority |
|----|------------|----------|
| FR-001 | Executor application form at `/pms/tasks/:id/duration-change` | Must |
| FR-002 | Date validation: proposed dates not before current date (mock) | Must |
| FR-003 | Reason field required with minimum length | Must |
| FR-004 | Leader review queue at `/pms/projects/:id/duration-requests` | Must |
| FR-005 | Approve/reject with opinion; approve updates task dates + audit log | Must |
| FR-006 | Gantt refresh on approve ([3.1.3.9](./3.1.3.9-gantt-chart.md)) | Should |
| FR-007 | Block request if task `completed` | Must |

### Edge Cases & Alternative Paths

- **Completed task:** Form unavailable; redirect with toast.
- **Pending request exists:** Show status banner; block duplicate submit.
- **Proposed end before proposed start:** Validation error.
- **Approve extends beyond project end:** Warning `Alert` to leader; allow with confirm.
- **Reject:** Executor sees opinion on `/pms/tasks/my` ([3.1.3.4](./3.1.3.4-project-progress-updates.md)).

## Phase 2: Screen-by-Screen Functional Specifications

### Screen: Duration Change Application — `/pms/tasks/:id/duration-change`

* **Objective:** Executor submits proposed schedule adjustment.

* **UI Elements & Field-Level Specs:**
  - `Breadcrumb`: My Tasks → `{Task Name}` → Duration Change.
  - Summary `Card`: current planned start/end (read-only), project name link.
  - `Form`:
    - **Adjusted planned start** — `Calendar`, required.
    - **Adjusted planned end** — `Calendar`, required; after start.
    - **Reason** — `Textarea`, required, min 20 chars, counter.
  - `Alert` info: dates cannot be before today (mock system date).
  - Actions: Cancel → `/pms/tasks/my`; Submit → confirmation toast + redirect.

* **Functional Logic:**
  1. Load task by `:id`; gate if `completed`.
  2. Validate dates ≥ today mock; create `DurationChangeRequestDto` status `pending`.
  3. Push to leader queue `/pms/projects/:id/duration-requests`.

---

### Screen: Leader Duration Review Queue — `/pms/projects/:id/duration-requests`

* **Objective:** Leader reviews pending duration change requests for a project.

* **UI Elements & Field-Level Specs:**
  - `Card` title "Duration Change Requests" with pending count `Badge`.
  - `DataTable`: Task Name, Executor, Original dates, Proposed dates, Reason (truncated), Submitted At, Actions (Review).
  - Tab on project detail or standalone route.

* **Functional Logic:**
  - Filter requests `Status === pending` for project.

---

### Modal: Review Duration Request — `Dialog`

* **Objective:** Leader approves or rejects a single request.

* **UI Elements & Field-Level Specs:**
  - Side-by-side date comparison `Card`s (original vs proposed).
  - Full reason `Textarea` read-only.
  - **Opinion** — `Textarea`, required on reject.
  - `RadioGroup`: Approve | Reject.
  - Submit decision `Button`.

* **Functional Logic:**
  1. Approve: update task `PlannedStart`/`PlannedEnd`; append audit log entry; refresh Gantt mock [3.1.3.9](./3.1.3.9-gantt-chart.md).
  2. Reject: status `rejected`; notify executor toast.

## Phase 3: Screen Transition & User Flow Mapping

| # | Trigger | Source | Destination | Conditional Paths |
|---|---------|--------|-------------|-------------------|
| 1 | "Request duration change" | `/pms/tasks/my` task row | `/pms/tasks/:id/duration-change` | Blocked if completed |

| 3 | Leader opens queue | Project detail | `/pms/projects/:id/duration-requests` | Leader permission |

| 5 | Approve | `Dialog` | Queue refresh | Task dates updated; Gantt [3.1.3.9](./3.1.3.9-gantt-chart.md) |
| 6 | Reject | `Dialog` | Queue refresh | Opinion to executor |
| 7 | Cancel | Application form | `/pms/tasks/my` | [3.1.3.4](./3.1.3.4-project-progress-updates.md) |
