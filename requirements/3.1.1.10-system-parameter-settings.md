# 3.1.1.10 Custom System Parameter Settings — Frontend Requirements

**Source:** F-BASE-010 — Section 3.1.1 System Basic Management  
**Route:** `/pms/admin/parameters`

---

## Phase 1: Deep Requirement Analysis

### User Personas

| Persona | Goals |
|---------|-------|
| **Supreme System Configuration Admin (primary)** | Tune global business rules (thresholds, toggles, dates) with correct data types and effective scheduling. |
| **Operations Director** | Understand when parameter changes take effect — immediately vs scheduled — before peak production periods. |
| **Auditor** | Review per-parameter change history: old value, new value, actor, timestamp. |
| **Module Runtime (indirect)** | Consumes `SystemParameter` values from `src/models/pms/configuration.ts` after effective time passes (mock). |

### Core Capabilities

1. **FR-001 — Parameter list:** **Table** at `/pms/admin/parameters` — Code, Name, Type, Current value, Effective at, Is core, Actions.
2. **FR-002 — List filtering:** Category filter **Select**; "Core only" **Switch** toggle.
3. **FR-003 — Type-aware value editor:** Edit **Dialog** renders control by `DataType` — string **Input**, number **Input**, boolean **Switch**, date **DatePicker**.
4. **FR-004 — Effective timing:** Radio — Immediately | Scheduled datetime; `EffectiveAt` must be future when scheduled.
5. **FR-005 — Core parameter safeguard:** `IsCore = true` requires double-confirmation **Dialog** before save.
6. **FR-006 — Value validation:** Value must match `DataType` per `SystemParameterSchema`.
7. **FR-007 — Per-parameter change log:** Tab or expandable row showing `ChangeLog` entries (mock).
8. **FR-008 — Supreme permission gate:** Only supreme system configuration permission.
9. **FR-009 — Async states:** Loading, empty, error, success for list load, edit save, and log fetch.

### Edge Cases & Alternative Paths

- Scheduled effective time in the past → validation error on save.
- User changes number value to non-numeric → inline Zod error.
- Boolean parameter displayed as **Switch** in list (read-only) and in editor.
- Core parameter: user cancels second confirmation → remain in edit **Dialog** with values preserved.
- Filter yields no parameters → empty **Table** state.
- Parameter code read-only in editor (identity key).
- User saves same value as current → Save disabled or warn "No changes."
- Double-click Save or core Confirm → disabled until request completes.
- Change log empty → "No changes recorded for this parameter."

## Phase 2: Screen-by-Screen Functional Specifications

### Screen Name: System Parameter List (`/pms/admin/parameters`)

* **Objective:** Browse and filter global parameters; open editor for value changes.

* **UI Elements & Field-Level Specs:**
  * **Page header:** "System Parameters"; supreme-config permission gate.
  * **Filter row:**
    * **Category (`Select`):** All + dynamic categories from mock data.
    * **Core only (`Switch`):** Filters `IsCore = true`.
    * **Search (`Input`):** Filter by code or name; debounced.
  * **Parameters `Table`:** Columns Code, Name, Category, DataType **Badge**, Value (formatted), EffectiveAt, Core **Badge**, Actions (Edit).
  * **Value column formatting:** Boolean → Yes/No; date → locale date; number → formatted.
  * **Row expand or View log link:** Opens change log panel (inline accordion or **Sheet**).
  * **Edit (`Button` ghost):** Opens Edit Parameter Dialog.
  * **Empty / loading / error states:** Standard.

* **Functional Logic:**
  * Fetch from `src/mocks/pms/system-parameters.ts`.
  * Filters client-side or mock server query.
  * Core rows visually distinct (icon or **Badge** variant).

### Screen Name: Edit Parameter Dialog (modal on `/pms/admin/parameters`)

* **Objective:** Change parameter value and effective timing with type-safe controls.

* **UI Elements & Field-Level Specs:**
  * **Dialog:** Title "Edit parameter — {Name}"; Code read-only subtitle.
  * **Core warning (`Alert` destructive):** Visible when `IsCore`; "Changes affect system-wide behavior."
  * **Current value (read-only):** Display formatted existing value.
  * **New value (dynamic by DataType):**
    * `string` → **Input** text max reasonable length.
    * `number` → **Input** type="number"; step per mock metadata.
    * `boolean` → **Switch** with Yes/No labels.
    * `date` → **DatePicker** / datetime per mock.
  * **Effective (`RadioGroup`):** Immediately (sets `EffectiveAt` to now mock) | Scheduled → **DateTimePicker** required when selected.
  * **Tabs or link — Change log:** Mini **Table**: At, Actor, OldValue, NewValue (last 10).
  * **Cancel (`Button` outline).**
  * **Save (`Button`):** Disabled when invalid, pristine, or submitting; core params open second confirm instead of direct save.

* **Functional Logic:**
  * Validate with `SystemParameterSchema` pick fields.
  * Non-core: Save → mock `PUT` → close → **Toast** "Parameter updated" → refresh list row.
  * Core: Save click → open Core Parameter Confirm Dialog (second step).
  * Scheduled: validate future `EffectiveAt`.
  * Append mock `ChangeLog` entry on success.

### Screen Name: Core Parameter Confirm Dialog (nested modal on `/pms/admin/parameters`)

* **Objective:** Double confirmation for high-impact core parameter changes.

* **UI Elements & Field-Level Specs:**
  * **Dialog:** Title "Confirm core parameter change"; lists parameter name, old → new value, effective timing.
  * **Acknowledgment checkbox:** "I understand this affects production behavior" — required to enable confirm.
  * **Cancel (`Button` outline):** Returns to Edit Parameter Dialog.
  * **Confirm save (`Button` destructive):** Disabled until checkbox checked.

* **Functional Logic:**
  * On confirm: same save path as non-core; **Toast** may include scheduled effective time.
  * On cancel: close only confirm dialog; edit dialog remains open.
  * Double-submit prevention.

### Screen Name: Parameter Change Log Sheet (optional `Sheet` on `/pms/admin/parameters`)

* **Objective:** Full history view when inline log exceeds preview length.

* **UI Elements & Field-Level Specs:**
  * **Sheet:** Title "Change history — {Code}"; **Table** with pagination.
  * **Close (`Button`).**

* **Functional Logic:**
  * Opened from "View full history" on list row or edit dialog.

## Phase 3: Screen Transition & User Flow Mapping

| # | Trigger Event | Source Screen | Destination Screen | Conditional Paths |
|---|---------------|---------------|--------------------|-------------------|
| 1 | Page load | — | System Parameter List | Non-supreme → 403 |
| 2 | Apply category / core filter | Parameter List | Parameter List (filtered) | — |
| 3 | Click **Edit** | Parameter List | Edit Parameter Dialog | — |
| 4 | Click **Save** (non-core, valid) | Edit Dialog | Parameter List | Dialog closes; Toast |
| 5 | Click **Save** (core, valid) | Edit Dialog | Core Parameter Confirm Dialog | — |
| 6 | Confirm core save | Core Confirm Dialog | Parameter List | Both dialogs close; Toast |
| 7 | Cancel core confirm | Core Confirm Dialog | Edit Parameter Dialog | Edit dialog stays open |
| 8 | Click **Cancel** | Edit Dialog | Parameter List | Dirty → confirm discard |
| 9 | Click **View full history** | Parameter List | Change Log Sheet | — |
| 10 | Close sheet | Change Log Sheet | Parameter List | — |
| 11 | Scheduled time invalid | Edit Dialog | Edit Dialog | Inline error; no save |
