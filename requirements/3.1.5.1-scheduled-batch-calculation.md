# 3.1.5.1 Scheduled Automatic Batch Calculation ÔÇö Frontend Requirements

**Source:** F-CALC-001 ÔÇö Section 3.1.5 KPI Calculation and Statistical Analysis  
**Route:** `/pms/kpi/calculation/jobs`, `/pms/kpi/calculation/jobs/:jobId`  
**UI:** shadcn/ui ÔÇö `Card`, `Table`, `DataTable`, `Badge`, `Button`, `Tabs`, `Progress`, `Alert`, `Tooltip`, `Sheet`, `Toast`, `Skeleton`

---

## Functional Summary

System automatically batch-calculates all corporate KPIs on preset cycles. Pulls approved data from collection module; flags anomalies and pushes admin alerts.

**Prerequisites:** Formulas configured; baseline data reviewed and stored.

---

## Phase 1: Deep Requirement Analysis

### User Personas

| Persona | Role | Goals | Pain points |
|---------|------|-------|-------------|
| **PMS Admin** | Calculation operator | Monitor job health, investigate failures, download logs | Silent failures until cockpit shows wrong numbers |
| **KPI Owner** | Indicator steward | See which indicators failed and why | No per-indicator error detail |
| **Executive** | Read-only consumer | Trust scheduled refresh timestamps | Stale data without visibility into last run |
| **System (mock scheduler)** | Background trigger | Fire jobs on cycle without UI block | N/A |

### Core Capabilities

| ID | Capability | Description |
|----|------------|-------------|
| FR-001 | Job list dashboard | Show all calculation jobs with cycle, schedule, last run, status |
| FR-002 | Job detail view | Per-indicator results: value, target, traffic-light status |
| FR-003 | Anomaly flagging | Rows with `anomaly` or `error` status highlighted; error message column |
| FR-004 | Error log access | Download / view error log for failed jobs (mock text file) |
| FR-005 | Admin alert on anomalies | Toast + mock alert record when job completes with anomalies |
| FR-006 | Running job indicator | In-progress jobs show `Progress` indeterminate + status `running` |
| FR-007 | Link to indicator config | Anomaly row links to [3.1.2.2](./3.1.2.2-kpi-indicator-lifecycle.md) |
| FR-008 | Async per-widget loading | List and detail sections load independently |

### Edge Cases & Alternative Paths

- **No approved data for period:** Job completes `partial`; empty result set with explanation.
- **All indicators fail:** Status `failed`; destructive banner on detail.
- **Job running when user opens detail:** Poll mock every 3s until terminal status.
- **Formula references missing indicator:** Per-row `error` with message.
- **Zero jobs configured:** Empty state with link to system parameters mock.
- **Anomaly threshold only (value computed):** Status `anomaly` not `error`; yellow row highlight.
- **User navigates away during poll:** Polling stops on unmount.
- **Duplicate job trigger (mock):** Second run queued; list shows both entries.

## Phase 2: Screen-by-Screen Functional Specifications

### Screen Name: Calculation Jobs Dashboard (`/pms/kpi/calculation/jobs`)

* **Objective:** Admin monitors all scheduled and recent batch calculation jobs.
* **UI Elements & Field-Level Specs:**
  * Page header: "KPI Calculation Jobs", subtitle showing next scheduled run (mock).
  * Summary `Card` row: Total jobs (24h), Success, Partial, Failed counts with semantic colors.
  * Filter bar: `Select` cycle, `Select` status, date range `DatePicker`.
  * `DataTable` columns: Job ID, Cycle, Scheduled at, Last run (duration subtitle), Status `Badge`, Indicators (total / success / anomaly), Actions (`Button` "View").
  * Status badges: Running (`secondary` + spinner), Success (`default`), Partial (`warning`), Failed (`destructive`).
  * Auto-refresh toggle (30s) for running jobs.
* **Functional Logic:**
  * Default sort: Scheduled at desc.
  * Click row Ôćĺ job detail.
  * On mock webhook of job complete with anomalies Ôćĺ global toast + link to detail.

### Screen Name: Job Detail (`/pms/kpi/calculation/jobs/:jobId`)

* **Objective:** Admin inspects per-indicator calculation outcomes for one job run.
* **UI Elements & Field-Level Specs:**
  * Breadcrumb: KPI Calculation > Jobs > `{JobId}`.
  * Header `Card`: cycle, started/completed timestamps, duration, overall status badge.
  * `Tabs`: Results | Error log.
  * **Results tab:**
    * Filter chips: All / OK / Anomaly / Error.
    * `Table`: Indicator name (link), Value, Target, Variance %, Status `Badge` (traffic-light from [3.1.6.3](./3.1.6.3-traffic-light-rules.md)), Error message (tooltip if truncated).
    * Anomaly/error rows: `bg-warning/5` or `bg-destructive/5`.
  * **Error log tab:**
    * `ScrollArea` monospace log text (mock).
    * `Button` "Download log" (mock blob).
  * Running state: `Progress` indeterminate + "Calculation in progressÔÇŽ".
* **Functional Logic:**
  * Fetch job; if `running`, poll until complete.
  * Indicator name click Ôćĺ `/pms/kpi/indicators/:id` mock or lifecycle doc route.
  * Anomaly count > 0 on load Ôćĺ `Alert` warning with count.

### Screen Name: Anomaly Alert Toast (Global)

* **Objective:** Notify admin immediately when batch job detects anomalies.
* **UI Elements & Field-Level Specs:**
  * shadcn `Toast` variant warning: "Batch job completed with {n} anomalies."
  * Action link: "View job" Ôćĺ detail route.
* **Functional Logic:**
  * Fires once per job completion mock event.
