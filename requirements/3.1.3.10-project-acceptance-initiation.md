# 3.1.3.10 Project Overall Acceptance Application Initiation — Frontend Requirements

**Source:** F-PROJ-010 — Section 3.1.3 Project Management  
**Route:** `/pms/projects/:id/acceptance`  
**Related:** [3.1.3.3](./3.1.3.3-sub-task-creation.md), [3.1.3.4](./3.1.3.4-project-progress-updates.md), [3.1.3.11](./3.1.3.11-project-acceptance-approval.md), [3.1.3.8](./3.1.3.8-kpi-data-sync.md)

---

## Functional Summary

When all sub-tasks pass acceptance, leader submits overall project completion acceptance to auditor.

**Prerequisites:** All sub-tasks fully accepted.

---

## Phase 1: Deep Requirement Analysis

### User Personas

| Persona | Role | Goals | Pain Points |
|---------|------|-------|-------------|
| **Project Leader** | Closes project formally | Submit when all tasks done | Submit blocked with unclear task gaps |
| **Auditor** | Reviews in [3.1.3.11](./3.1.3.11-project-acceptance-approval.md) | Receive complete outcome package | Premature submissions |
| **Task Executor** | Completes sub-tasks | Understand acceptance criteria per [3.1.3.3](./3.1.3.3-sub-task-creation.md) | Tasks stuck below 100% |

### Core Capabilities (FR-xxx)

| ID | Capability | Priority |
|----|------------|----------|
| FR-001 | Pre-check gate banner: N of N tasks accepted | Must |
| FR-002 | Submit disabled until 100% tasks accepted | Must |
| FR-003 | Acceptance application form with summary, target achievement, attachments | Must |
| FR-004 | On submit: status `pending_acceptance`; auditor queue +1 mock | Must |
| FR-005 | Task acceptance checklist read-only table | Should |
| FR-006 | KPI achievement preview from [3.1.3.8](./3.1.3.8-kpi-data-sync.md) | Should |

### Edge Cases & Alternative Paths

- **Incomplete tasks:** Banner lists remaining tasks with links to [3.1.3.4](./3.1.3.4-project-progress-updates.md).
- **Resubmit after fail:** Form pre-filled from prior application [3.1.3.11](./3.1.3.11-project-acceptance-approval.md).
- **Missing attachments:** Submit blocked.
- **Project not `in_progress`:** Route guard with explanation.
- **Already pending acceptance:** Read-only submitted state with submitted timestamp.

## Phase 2: Screen-by-Screen Functional Specifications

### Screen: Acceptance Application — `/pms/projects/:id/acceptance`

* **Objective:** Leader submits overall project acceptance when all tasks complete.

* **UI Elements & Field-Level Specs:**
  - `Breadcrumb`: Project → Acceptance Application.
  - **Pre-check gate** — `Alert` variant success when all accepted; variant warning when not:
    - Text: "{accepted} of {total} tasks accepted".
    - `Progress` bar visual.
    - If incomplete: `DataTable` of pending tasks with Owner, Progress %, link to task.
  - **Task checklist** — collapsible `Card` all tasks with acceptance criteria excerpt and status `Badge`.
  - **Application form** `Card` (enabled only when 100%):
    - **Overall completion summary** — `Textarea`, required, min 50 chars.
    - **Target achievement status** — `RadioGroup`: met | partial | not_met.
    - **Outcome attachments** — file `Input` mock multi-upload, required ≥ 1 file.
  - **KPI preview** — read-only mini table from [3.1.3.8](./3.1.3.8-kpi-data-sync.md): KPI name, synced value, target.
  - Footer: `Button` "Submit for Acceptance" (primary, disabled until gate pass).

* **Functional Logic:**
  1. Compute `AllTasksAccepted` from task statuses (all `completed` with acceptance mock).
  2. Submit creates `ProjectAcceptanceApplicationDto`; project status `pending_acceptance`.
  3. Mock auditor queue increment [3.1.3.11](./3.1.3.11-project-acceptance-approval.md); toast success.
  4. Redirect to project detail with pending banner.

---

### Banner: Pending Acceptance — project detail

* **Objective:** Show submitted state awaiting auditor.

* **UI Elements & Field-Level Specs:** `Alert` info: "Acceptance pending review" + submitted date.

* **Functional Logic:** Shown after successful submit until [3.1.3.11](./3.1.3.11-project-acceptance-approval.md) decision.

- [ ] Resubmit flow after auditor fail with pre-filled data
- [ ] Pending acceptance banner on project detail post-submit
- [ ] Four async states: loading, incomplete tasks warning, validation error, success toast
