# 3.1.9.2 Implementation & Execution Tracking of Improvement Proposals — Frontend Requirements

**Source:** F-PDCA-002 — Section 3.1.9 PDCA Improvement Closed-Loop Management  
**Routes:** `/pms/pdca/proposals/:id/execution`, `/pms/pdca/tasks`  
**Related workflow:** Follows approved proposals from [3.1.9.1](./3.1.9.1-pdca-proposal-submission.md); may link from [3.1.8.6](./3.1.8.6-secondary-confirmation.md) when HR-generated PDCA is in progress

---

## Functional Summary

Approved proposals decompose into execution tasks with owners and deadlines. Assignees update progress in real time; management monitors the full rectification lifecycle.

**Input:** Proposal ID, execution steps, responsible persons, planned completion deadlines.  
**Prerequisites:** Proposal reviewed and approved.

---

## Phase 1: Deep Requirement Analysis

### User Personas

| Persona | Goals | Pain Points |
|---------|-------|-------------|
| **Manager / Auditor** | Break approved proposals into owned steps with deadlines | No visibility into execution after approval |
| **Task Assignee (Employee)** | Update progress on assigned steps | Tasks buried in email |
| **HR / Executive** (monitor) | See overdue steps and aggregate progress | Disconnected from [3.1.8.5](./3.1.8.5-hr-rectification.md) outcomes |

### Core Capabilities (FR-001…)

| ID | Capability | Priority |
|----|------------|----------|
| FR-001 | Management task decomposition UI on approved proposal | Must |
| FR-002 | Dynamic execution steps: description, owner, deadline, order | Must |
| FR-003 | **Create execution tasks** → assignee to-dos; proposal → `In execution` | Must |
| FR-004 | Assignee personal task list at `/pms/pdca/tasks` | Must |
| FR-005 | Progress update: %, notes (required if <100%), optional evidence | Must |
| FR-006 | Overdue visual flag when past deadline | Must |
| FR-007 | Management monitor: timeline/Gantt mock, aggregate progress | Must |
| FR-008 | Optional alert cross-link [3.1.7.1](./3.1.7.1-alert-rule-configuration.md) for overdue | Should |

### Edge Cases & Alternative Paths

- **Proposal not approved:** Execution page 403 or redirect to [3.1.9.1](./3.1.9.1-pdca-proposal-submission.md) detail.
- **Zero steps on create:** Block **Create execution tasks**; min 1 step.
- **Duplicate owner on multiple steps:** Allowed.
- **Progress 100% without notes:** Allowed on final completion.
- **Progress <100% save without notes:** Block with inline error.
- **Deadline in past on create:** Warning; allow with acknowledgment.
- **All tasks complete:** Proposal status → `completed` (mock).
- **Assignee views another user's task:** 403 on direct URL.
- **Refresh sync:** Manual refresh button; mock polling optional.

## Phase 2: Screen-by-Screen Functional Specifications

### Screen: Task Decomposition — Management (`/pms/pdca/proposals/:id/execution`)

* **Objective:** Decompose approved proposal into ordered execution tasks and monitor progress.
* **UI Elements & Field-Level Specs:**
  * Breadcrumb: Proposals → {Title} → Execution
  * Proposal header: title, category, link back to [3.1.9.1](./3.1.9.1-pdca-proposal-submission.md) detail
  * Visible only when status `Approved` or `In execution` (management permission)
  * **Execution steps editor** (when not yet created or editable draft):
    * Dynamic rows: Step description (required), Responsible person (employee picker, required), Planned deadline (date, required), Order (drag handle)
    * **Add step**, **Remove step**
    * **Create execution tasks** primary (disabled if 0 rows or invalid)
  * **Management monitor** (after tasks created):
    * Overall progress bar (% = average of step progress)
    * Gantt-style or stepper timeline (mock): per-step owner, deadline, progress %, overdue flag
    * Alert badge when any overdue — link to [3.1.7.1](./3.1.7.1-alert-rule-configuration.md) (optional mock)
  * Four async states
* **Functional Logic:**
  * Create tasks: generate `PdcaExecutionTaskDto` per row; notify assignees (mock); status → `in_execution`
  * Recompute aggregate on assignee updates (mock refresh)

### Screen: Assignee To-Do List (`/pms/pdca/tasks`)

* **Objective:** Personal list of PDCA execution tasks assigned to current user.
* **UI Elements & Field-Level Specs:**
  * Page title: "My PDCA Tasks"
  * Table: Proposal title (link to execution page), Step description, Deadline (red if overdue), Progress %, Status badge, Actions (**Update progress**)
  * Status meanings: Not started, In progress, Completed, Overdue
  * Filters: Status, Overdue only
  * Empty state: "No tasks assigned"
  * Four async states
* **Functional Logic:**
  * Scope to `OwnerId === currentUser`
  * Overdue computed: deadline < today && status !== completed

### Modal: Progress Update (Assignee)

* **Objective:** Assignee reports step progress.
* **UI Elements & Field-Level Specs:**
  * Step description (read-only), Proposal title (read-only)
  * Progress % (slider 0–100 or numeric input)
  * Progress notes (textarea; required if progress < 100 on save)
  * Completion evidence (optional mock attachment)
  * **Save** / **Cancel**
* **Functional Logic:**
  * Append to `ProgressNotes` history with timestamp
  * Update parent proposal aggregate progress
  * At 100%: status → completed unless overdue still flagged until saved

### Screen: Execution Monitor (embedded on execution page post-create)

* **Objective:** Management view of all steps for one proposal.
* **UI Elements & Field-Level Specs:**
  * Same as monitor section above — stepper with expand per step showing notes history
  * **Refresh** button for mock sync
* **Functional Logic:**
  * Read-only for non-managers on management fields; assignees use tasks modal
