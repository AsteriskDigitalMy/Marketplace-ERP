# 3.1.3.2 Project Initiation Workflow Approval — Frontend Requirements

**Source:** F-PROJ-002 — Section 3.1.3 Project Management  
**Route:** `/pms/projects/approvals`  
**Related:** [3.1.3.1](./3.1.3.1-project-initiation.md) (initiation), [3.1.3.3](./3.1.3.3-sub-task-creation.md) (post-approval tasks), [3.1.3.9](./3.1.3.9-gantt-chart.md) (Gantt init), [3.1.1.7](./3.1.1.7-user-role-binding.md) (auditor role)

## Functional Summary

Auditors approve or reject project initiation applications. Pass → `In Progress` + Gantt sync; reject → return to initiator with required reason.

**Prerequisites:** Project approval permission; status `Pending Approval`.

---

## Phase 1: Deep Requirement Analysis

### User Personas

| Persona | Role | Goals | Pain Points |
|---------|------|-------|-------------|
| **Project Auditor** | PMO / executive approver | Fast review of complete packages | Missing context on KPI bindings |
| **Project Leader** | Submitter ([3.1.3.1](./3.1.3.1-project-initiation.md)) | Timely approval notification | Opaque rejection reasons |
| **Operations Manager** | Tracks approved portfolio | See budget and team allocation | No visibility until approved |

### Core Capabilities (FR-xxx)

| ID | Capability | Priority |
|----|------------|----------|
| FR-001 | Approval inbox table at `/pms/projects/approvals` | Must |
| FR-002 | Review detail with read-only initiation materials | Must |
| FR-003 | Decision: Approve / Reject with opinion required on reject | Must |
| FR-004 | Approve → status `in_progress`; Gantt initialized mock ([3.1.3.9](./3.1.3.9-gantt-chart.md)) | Must |
| FR-005 | Reject → status `returned`; leader edit CTA | Must |
| FR-006 | Notification toast to project leader on decision (mock) | Must |
| FR-007 | Inbox badge count in nav shell | Should |

### Edge Cases & Alternative Paths

- **Empty inbox:** Empty state "No pending approvals".
- **Approve already processed (stale):** Error toast "Already reviewed"; refresh list.
- **Reject without opinion:** `FormMessage` required field block.
- **Concurrent auditors:** First decision wins; second gets conflict toast.
- **Leader views returned project:** Banner with rejection opinion + "Edit application" → [3.1.3.1](./3.1.3.1-project-initiation.md).

## Phase 2: Screen-by-Screen Functional Specifications

### Screen: Approval Inbox — `/pms/projects/approvals`

* **Objective:** List all `pending_approval` projects awaiting auditor action.

* **UI Elements & Field-Level Specs:**
  - `Card` title "Project Initiation Approvals" with count `Badge`.
  - `DataTable`: Project Code, Name, Leader (avatar + name), Budget (formatted), Submitted At, Actions (`Button` "Review").
  - `Input` search by code/name; sort by submitted date desc default.
  - Skeleton loading; empty state illustration.

* **Functional Logic:**
  - Filter mock projects where `Status === pending_approval`.
  - Row "Review" → `/pms/projects/approvals/:projectId`.

---

### Screen: Review Detail — `/pms/projects/approvals/:projectId`

* **Objective:** Present full initiation package and capture approve/reject decision.

* **UI Elements & Field-Level Specs:**
  - `Breadcrumb`: Approvals → `{Project Code}`.
  - Read-only `Card` sections mirroring [3.1.3.1](./3.1.3.1-project-initiation.md): Basic Info, Team, Budget, Client, Gantt sketch.
  - **Bound KPI list** — `DataTable` Code, Name, Target; links to [3.1.2.2](./3.1.2.2-kpi-indicator-lifecycle.md).
  - **Budget summary** — `Card` with amount highlight.
  - Decision `Form` (sticky footer on mobile):
    - **Decision** — `RadioGroup`: Approve | Reject.
    - **Opinion** — `Textarea`, required when Reject, optional on Approve.
  - Actions: `Button` "Back"; primary `Button` "Submit Decision".

* **Functional Logic:**
  1. Approve: update status `in_progress`; mock Gantt init ([3.1.3.9](./3.1.3.9-gantt-chart.md)); toast to leader "Project approved"; redirect to inbox.
  2. Reject: status `returned`; opinion stored; toast to leader with edit link; redirect to inbox.
  3. Enable task creation path [3.1.3.3](./3.1.3.3-sub-task-creation.md) only after approve.

---

### Modal: Confirm Approve — `AlertDialog`

* **Objective:** Confirm budget and KPI commitments before approval.

* **UI Elements & Field-Level Specs:** Summary text; Confirm / Cancel.

* **Functional Logic:** Optional for high-budget threshold mock (> 1M).
