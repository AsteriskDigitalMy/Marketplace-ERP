# 3.1.2.1 New KPI Indicator Addition — Frontend Requirements

**Source:** F-KPI-001 — Section 3.1.2 KPI Indicator Library Management  
**Route:** `/pms/kpi/indicators/new`  
**Related:** [3.1.2.2](./3.1.2.2-kpi-indicator-lifecycle.md) (lifecycle), [3.1.2.3](./3.1.2.3-kpi-formula-editor.md) (formula editor), [3.1.1.8](./3.1.1.8-global-dictionary-maintenance.md) (dictionaries), [3.1.5.1](./3.1.5.1-scheduled-batch-calculation.md) (calculation)

---

## Functional Summary

Create performance evaluation indicators with formula, targets, cycle, data source, weight, and alert threshold. Saving creates version V1.0 after formula simulation validation.

**Prerequisites:** KPI indicator library management permission.

---

## Phase 1: Deep Requirement Analysis

### User Personas

| Persona | Role | Goals | Pain Points |
|---------|------|-------|-------------|
| **KPI Administrator** | HR/Operations analyst with `kpi:indicator:manage` | Define measurable indicators aligned to leather-industry standards | Formula errors discovered only after go-live; duplicate codes across plants |
| **Department Manager** | Read-only consumer of indicators | Understand what will be measured before project binding | Opaque formula definitions without trial output |
| **System Integrator** | Technical configurator | Ensure indicators reference valid data sources for [3.1.4](./3.1.4.1-periodic-filling-tasks.md) collection | Mixed auto/manual sources unclear at creation time |

### Core Capabilities (FR-xxx)

| ID | Capability | Priority |
|----|------------|----------|
| FR-001 | Render full indicator attribute form on `/pms/kpi/indicators/new` | Must |
| FR-002 | Enforce globally unique indicator code (async debounced check) | Must |
| FR-003 | Embed formula editor per [3.1.2.3](./3.1.2.3-kpi-formula-editor.md) with mandatory pre-save simulation | Must |
| FR-004 | Validate target value > 0; optional challenge/baseline numeric bounds | Must |
| FR-005 | Dictionary-driven business category picker from [3.1.1.8](./3.1.1.8-global-dictionary-maintenance.md) | Must |
| FR-006 | On successful save, persist `Version: V1.0`, `Status: enabled`, toast confirmation | Must |
| FR-007 | Cancel with unsaved-changes `AlertDialog` guard | Should |
| FR-008 | Breadcrumb navigation: KPI Library → New Indicator | Should |

### Edge Cases & Alternative Paths

- **Duplicate code:** Inline `FormMessage` under code field; submit blocked until resolved.
- **Formula simulation failure:** Save button disabled; validation panel shows errors from [3.1.2.3](./3.1.2.3-kpi-formula-editor.md).
- **Network timeout on uniqueness check:** Show retry affordance; do not allow optimistic submit.
- **Partial form completion + navigate away:** `AlertDialog` "Discard unsaved changes?"
- **Data source = manual:** Show info `Alert` linking to [3.1.4.2](./3.1.4.2-manual-data-entry.md) for future filling tasks.
- **Empty dictionary category list:** Empty state with link to dictionary maintenance.
- **Permission denied:** Redirect to `/pms/kpi/indicators` with destructive toast.

## Phase 2: Screen-by-Screen Functional Specifications

### Screen: KPI Indicator List (entry point) — `/pms/kpi/indicators`

* **Objective:** Browse existing indicators and launch creation flow.

* **UI Elements & Field-Level Specs:**
  - `Card` page header with title "KPI Indicator Library" and primary `Button` "New Indicator" → `/pms/kpi/indicators/new`.
  - `DataTable` columns: Code, Name, Category (`Badge`), Cycle, Status (`Badge` enabled/disabled), Version, Actions (View → [3.1.2.2](./3.1.2.2-kpi-indicator-lifecycle.md)).
  - `Input` search filter (code/name); `Select` filters for category and status.
  - Skeleton rows (loading); empty state with CTA to create first indicator.

* **Functional Logic:**
  - Load mock list from `src/mocks/pms/kpi-indicators.ts`.
  - "New Indicator" visible only with manage permission.

---

### Screen: New KPI Indicator Form — `/pms/kpi/indicators/new`

* **Objective:** Capture all indicator attributes and create V1.0 after validated formula simulation.

* **UI Elements & Field-Level Specs:**
  - Layout: `Card` with `Form` (react-hook-form + zod). Two-column grid on `lg+`, single column on mobile.
  - **Indicator code** — `Input`, required, pattern `[A-Z0-9_]{3,32}`, debounced uniqueness API mock; `Loader2` icon while checking.
  - **Indicator name** — `Input`, required, max 128 chars.
  - **Business category** — `Select` / `Combobox` from dictionary `kpi_category`.
  - **Statistical scope/method** — `Textarea`, required, max 500 chars.
  - **Calculation formula** — embedded panel from [3.1.2.3](./3.1.2.3-kpi-formula-editor.md); `Sheet` or inline `Card` with "Open formula editor" link to `/pms/kpi/indicators/new/formula` (modal route optional).
  - **Target value** — `Input` type number, required, min exclusive 0, step 0.01.
  - **Challenge value** — `Input` number, optional; must be > target if provided.
  - **Baseline value** — `Input` number, optional.
  - **Statistical cycle** — `Select`: daily | weekly | monthly | quarterly | annual.
  - **Evaluation object** — `RadioGroup`: department | individual | project.
  - **Data source** — `Select`: auto | manual | mixed; contextual `Alert` for manual/mixed.
  - **Evaluation weight %** — `Input` number 0–100, optional.
  - **Anomaly alert threshold** — `Input` number, optional; `Tooltip` explains linkage to [3.1.7.1](./3.1.7.1-alert-rule-configuration.md).
  - Footer: `Button` variant outline "Cancel"; `Button` variant secondary "Simulate & Validate"; primary `Button` "Save Indicator" (disabled until simulation pass).

* **Functional Logic:**
  1. "Simulate & Validate" invokes formula validation mock; renders result `Card` (pass/fail, simulated output).
  2. Save blocked if: validation not passed, code duplicate, target ≤ 0, or required fields empty.
  3. On save success: toast `"Indicator created — Version V1.0"`; navigate to `/pms/kpi/indicators/:id` ([3.1.2.2](./3.1.2.2-kpi-indicator-lifecycle.md)).
  4. All four async states: skeleton form (loading), validation error `Alert`, empty dictionaries message, success toast.

---

### Modal: Unsaved Changes Guard — `AlertDialog`

* **Objective:** Prevent accidental data loss on navigation.

* **UI Elements & Field-Level Specs:** `AlertDialog` title "Discard changes?"; actions "Stay" / "Discard".

* **Functional Logic:** Triggered on route change or Cancel when `formState.isDirty`.

## Phase 3: Screen Transition & User Flow Mapping

| # | Trigger | Source | Destination | Conditional Paths |
|---|---------|--------|-------------|-------------------|
| 1 | Click "New Indicator" | `/pms/kpi/indicators` | `/pms/kpi/indicators/new` | Hidden if no permission |
| 2 | Click "Simulate & Validate" | New Indicator Form | Same (result panel) | Fail → errors shown; pass → enable Save |
| 3 | Click "Open formula editor" | New Indicator Form | Formula editor embed / `/pms/kpi/indicators/new/formula` | Returns formula string to parent form |
| 4 | Click "Save Indicator" | New Indicator Form | `/pms/kpi/indicators/:id` | Blocked until validation pass |
| 5 | Click "Cancel" | New Indicator Form | `/pms/kpi/indicators` | Dirty → `AlertDialog` confirm |
| 6 | Breadcrumb "KPI Library" | New Indicator Form | `/pms/kpi/indicators` | Dirty → `AlertDialog` |
| 7 | Permission denied on load | Any | `/pms/kpi/indicators` | Toast error |
