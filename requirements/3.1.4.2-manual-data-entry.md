# 3.1.4.2 Manual Form Entry for Business Data — Frontend Requirements

**Source:** F-DATA-002 — Section 3.1.4 Data Collection and Filling  
**Route:** `/pms/data-collection/fill/:taskId`  
**UI:** shadcn/ui — `Card`, `Form`, `Input`, `Textarea`, `Select`, `DatePicker`, `Button`, `Badge`, `Alert`, `Separator`, `Toast`, `Skeleton`

---

## Functional Summary

Employees complete standardized forms for production/business data. Real-time validation; submit → multi-level approval queue, status `Pending Review`.

**Prerequisites:** Employee has received filling task ([3.1.4.1](./3.1.4.1-periodic-filling-tasks.md)).

---

## Phase 1: Deep Requirement Analysis

### User Personas

| Persona | Role | Goals | Pain points |
|---------|------|-------|-------------|
| **Data Filler** | Task assignee | Enter accurate values quickly with clear validation feedback | Unclear which fields are wrong until submit |
| **Reviewer (downstream)** | Team / dept lead | Receive complete, valid submissions | Rework from missing or out-of-range values |
| **PMS Admin** | Configuration owner | Ensure template fields map to KPI thresholds | Threshold changes not reflected in form |

### Core Capabilities

| ID | Capability | Description |
|----|------------|-------------|
| FR-001 | Template-driven form render | Dynamic fields from mock schema: number, text, date, select |
| FR-002 | Real-time validation | Required, min/max, dictionary options validated on blur and change |
| FR-003 | KPI threshold hints | Inline helper text showing acceptable range from indicator config |
| FR-004 | Draft persistence | Optional "Save draft" stores `Status: draft` without entering review queue |
| FR-005 | Submit to review | Valid submit → `pending_review`; supervisor queue +1 (mock) |
| FR-006 | Read-only mode | Submitted / approved / rejected tasks open form read-only with status banner |
| FR-007 | Rejection resubmit | Rejected tasks show reviewer opinion `Alert`; editable fields re-enabled |
| FR-008 | Async state coverage | Loading schema, saving draft, submitting — distinct loading indicators |

### Edge Cases & Alternative Paths

- **Invalid or missing taskId:** 404 page with link back to my-tasks.
- **Task already approved:** Full read-only; no Save/Submit buttons.
- **Partial draft on navigate away:** `AlertDialog` unsaved changes warning.
- **Select field with empty dictionary:** Show inline error "Options unavailable" + disable submit.
- **Number field scientific notation:** Normalize to decimal on blur.
- **Concurrent draft save:** Last-write-wins mock; toast on conflict optional.
- **Submit with network error:** Form values retained; retry submit button.
- **Rejected task:** Pre-fill prior values + show `Alert` destructive with `Opinion` from [3.1.4.3](./3.1.4.3-multi-level-data-review.md).

## Phase 2: Screen-by-Screen Functional Specifications

### Screen Name: Dynamic Filling Form (`/pms/data-collection/fill/:taskId`)

* **Objective:** Assignee completes all template fields for one filling task and submits for review.
* **UI Elements & Field-Level Specs:**
  * Breadcrumb: Data Collection > My Tasks > Fill `{IndicatorName}`.
  * Header `Card`:
    * Indicator name (h1), period label `Badge`, due date with countdown.
    * Status `Badge`: Draft / Pending Review / Approved / Rejected.
    * Rejected: `Alert` variant destructive showing reviewer opinion.
  * `Form` body — one `FormField` per template field:
    * `number` — `Input` type number, min/max attrs, unit suffix if provided.
    * `text` — `Input` or `Textarea` based on max length mock.
    * `date` — `DatePicker` (popover + calendar).
    * `select` — `Select` with dictionary options.
    * Each field: label, required asterisk, description/helper, `FormMessage` inline error.
  * Sticky footer bar: "Save draft" (`Button` outline), "Submit for review" (`Button` primary), "Back to my tasks" (`Button` ghost).
  * Validation summary `Alert` at top if submit blocked (list of field errors).
* **Functional Logic:**
  * Load `DataFillingFormDto` by `taskId`; skeleton while loading.
  * Validate on blur and on submit; block submit until all valid.
  * Save draft → PATCH mock → toast "Draft saved" → status `draft`.
  * Submit → POST mock → toast success → status `pending_review` → disable editing → redirect option to my-tasks.
  * Approved / pending_review (non-owner): read-only all inputs.

### Screen Name: Form Loading / Error States (same route)

* **Objective:** Handle fetch failures without losing navigation context.
* **UI Elements & Field-Level Specs:**
  * Loading: header skeleton + 6 field skeletons.
  * Error: `Alert` with message + "Retry" `Button`.
  * Empty/missing template: "Template not found" empty state.
* **Functional Logic:**
  * Retry re-fetches form schema and values.

## Phase 3: Screen Transition & User Flow Mapping

| # | Trigger | Source | Destination | Conditional paths |
|---|---------|--------|-------------|-------------------|
| 1 | Click "Fill" on task | `/pms/data-collection/my-tasks` | `/pms/data-collection/fill/:taskId` | — |
| 2 | Save draft | Fill form | Same page | Validation optional for draft |
| 3 | Submit for review | Fill form | Same page (read-only) or my-tasks | Validation fail → stay, show errors |
| 4 | Back to my tasks | Fill form footer | `/pms/data-collection/my-tasks` | Unsaved changes → confirm dialog |
| 5 | Rejected notification click | Message center mock | Fill form (editable) | Pre-fill + opinion banner |
| 6 | Approved task open | My tasks "View" | Fill form read-only | — |
| 7 | Submit success | Fill form | `/pms/data-collection/reviews` queue (reviewer) | Reviewer not notified in filler flow |
| 8 | Invalid taskId | Direct URL | 404 / redirect my-tasks | — |

## Acceptance Criteria

- [ ] Template-driven dynamic form with all four field types
- [ ] Real-time threshold and required validation with inline errors
- [ ] Submit enters Pending Review and locks editing
- [ ] Draft save without entering review queue
- [ ] Rejected task shows opinion and allows resubmit
- [ ] Read-only mode for submitted/approved records
- [ ] Unsaved changes warning on navigate away
- [ ] Four async states (loading, empty, error, success)
