# 3.1.3.3 Sub-task Creation and Personnel Assignment — Frontend Requirements

**Source:** F-PROJ-003 — Section 3.1.3 Project Management  
**Route:** `/pms/projects/:id/tasks`  
**Related:** [3.1.3.1](./3.1.3.1-project-initiation.md), [3.1.3.2](./3.1.3.2-project-initiation-approval.md), [3.1.3.4](./3.1.3.4-project-progress-updates.md) (progress), [3.1.3.9](./3.1.3.9-gantt-chart.md) (Gantt)

---

## Functional Summary

Decompose projects into sub-tasks with owners, duration, acceptance criteria, update cycle, and prerequisite dependencies. Created tasks push to executor to-do lists.

**Prerequisites:** Project `In Progress`; task assignment permission.

---

## Phase 1: Deep Requirement Analysis

### User Personas

| Persona | Role | Goals | Pain Points |
|---------|------|-------|-------------|
| **Project Leader** | Decomposes work | Assign clear owners and dependencies | Circular dependencies break scheduling |
| **Task Executor** | Receives assignments | See tasks in personal queue ([3.1.3.4](./3.1.3.4-project-progress-updates.md)) | Unclear acceptance criteria |
| **PMO Viewer** | Read-only oversight | Understand WBS structure | No Gantt visibility until tasks exist |

### Core Capabilities (FR-xxx)

| ID | Capability | Priority |
|----|------------|----------|
| FR-001 | Task list tab on project detail `/pms/projects/:id/tasks` | Must |
| FR-002 | Create sub-task via `Dialog` with all required fields | Must |
| FR-003 | Prerequisite multi-select with cycle detection (mock graph) | Must |
| FR-004 | Duration validation within project planned span | Must |
| FR-005 | Executor required; pushes to executor to-do mock (+1 badge) | Must |
| FR-006 | Task CRUD: edit, delete (if not started) | Should |
| FR-007 | Task list feeds Gantt generation [3.1.3.9](./3.1.3.9-gantt-chart.md) | Must |

### Edge Cases & Alternative Paths

- **Project not `in_progress`:** Task tab read-only; create disabled with `Tooltip`.
- **Circular dependency selected:** Inline error; save blocked.
- **Planned end beyond project end:** Validation error.
- **Delete task with dependents:** Block delete; list dependent task names.
- **Executor not on project team:** Warning `Alert`; allow with confirm.
- **Zero tasks:** Empty state CTA "Create first sub-task".

## Phase 2: Screen-by-Screen Functional Specifications

### Screen: Project Detail — Tasks Tab — `/pms/projects/:id/tasks`

* **Objective:** Manage sub-tasks for an in-progress project.

* **UI Elements & Field-Level Specs:**
  - Project header `Card` (from parent layout): name, code, status, date range.
  - `Tabs` on project detail: Overview | **Tasks** | Progress | Gantt | Issues | KPI Sync (cross-links to [3.1.3.4](./3.1.3.4-project-progress-updates.md), [3.1.3.9](./3.1.3.9-gantt-chart.md), [3.1.3.7](./3.1.3.7-issue-closed-loop.md), [3.1.3.8](./3.1.3.8-kpi-data-sync.md)).
  - `Button` "New Sub-task" (primary) opens `Dialog`.
  - `DataTable`: Task Name, Owner, Planned Duration (start–end), Dependencies (count `Badge`), Status (`Badge`), Progress %, Actions (Edit, Delete).

* **Functional Logic:**
  - Load tasks from `src/mocks/pms/project-tasks.ts` filtered by `ProjectId`.
  - Create opens dialog; success refreshes table and mock Gantt source.

---

### Modal: New / Edit Sub-task — `Dialog`

* **Objective:** Capture task definition and assignment.

* **UI Elements & Field-Level Specs:**
  - `Form` fields:
    - **Task name** — `Input`, required.
    - **Executor** — `Combobox` user picker, required; default project team list.
    - **Planned start / end** — `Calendar` pickers; duration ≤ project span.
    - **Workload** — `Input` number hours, optional.
    - **Acceptance criteria** — `Textarea`, required.
    - **Update cycle** — `Select`: daily | weekly ([3.1.3.4](./3.1.3.4-project-progress-updates.md)).
    - **Prerequisites** — multi-select `Combobox` of sibling tasks; cycle check on change.
  - Footer: Cancel | Save.

* **Functional Logic:**
  1. On save: create `ProjectSubTaskDto`; status `not_started`; progress 0.
  2. Mock push to executor `/pms/tasks/my` badge +1 ([3.1.3.4](./3.1.3.4-project-progress-updates.md)).
  3. Edit mode pre-fills; delete only if `not_started` and no dependents.

---

### Drawer: Task Quick View — `Sheet`

* **Objective:** Read-only task summary from table row click.

* **UI Elements & Field-Level Specs:** All fields read-only; link "Update progress" if executor.

* **Functional Logic:** Non-blocking inspection; optional entry to progress flow.
