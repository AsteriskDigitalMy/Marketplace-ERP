# 3.1.6.1 Role-Specific Cockpit Automatic Loading — Frontend Requirements

**Source:** F-DASH-001 — Section 3.1.6 KPI Visualization Monitoring Dashboard  
**Route:** `/pms/cockpit`  
**Related:** [3.7.2.6 KPI/PMS BI dashboard](./3.7.2.6-kpi-pms-dashboard.md) (read-only analytics layer)  
**UI:** shadcn/ui — `Card`, `Badge`, `Button`, `Select`, `Table`, `Skeleton`, `Alert`, `Chart` (Recharts), `Tooltip`, `DropdownMenu`, `EmptyState`

---

## Functional Summary

Automatically matches and displays the exclusive visual data workbench for the logged-in user's role. Chart data loads asynchronously via backend APIs with scheduled auto-refresh.

**Prerequisites:** User is authenticated.  
**Input:** System role of the current user (from session/mock auth context).

---

## Phase 1: Deep Requirement Analysis

### User Personas

| Persona | Role | Goals | Pain points |
|---------|------|-------|-------------|
| **Executive / CFO** | Strategic leader | One-glance group KPI health and alert summary | Wrong dashboard for role; stale numbers |
| **Department Manager** | Line manager | Dept KPI cards, team tables, progress trends | Too much corporate noise |
| **Auditor** | Compliance reviewer | Pending review counts, exception heatmap | Missing operational drill path |
| **HR Manager** | People ops | Low-performance queue, PDCA pipeline | Disconnected from KPI cockpit |
| **Employee** | Individual contributor | Personal KPI progress, appraisal status | Overwhelming exec widgets |

### Core Capabilities

| ID | Capability | Description |
|----|------------|-------------|
| FR-001 | Role → layout auto-match | On load, resolve mock role and render matching cockpit variant |
| FR-002 | Summary cards row | 4–6 role-specific KPI cards with value, unit, trend, traffic-light color |
| FR-003 | Statistical charts grid | 2–4 charts (bar, line, donut) with independent async load |
| FR-004 | Business data table | Sortable, paginated role-relevant records; row click → drill-down |
| FR-005 | Manual refresh | Header button re-fetches all widgets |
| FR-006 | Auto-refresh interval | Selector: 30s / 1m / 5m / off; preserves scroll on refresh |
| FR-007 | Per-widget error isolation | Failed widget shows inline retry; others remain |
| FR-008 | Traffic-light integration | Card/chart colors from [3.1.6.3](./3.1.6.3-traffic-light-rules.md) via `getStatusColor()` |
| FR-009 | Unsupported role empty state | Friendly message + contact admin CTA |

### Edge Cases & Alternative Paths

- **Unknown or guest role:** Empty state; no widgets fetched.
- **Partial widget failure:** Skeleton → error card with retry for failed only.
- **Auto-refresh during drill-down sheet open:** Parent cockpit refreshes; sheet data unchanged until closed.
- **Role changed mid-session (mock):** Full layout swap on next navigation to `/pms/cockpit`.
- **Zero data for role:** Empty states per widget, not whole page error.
- **Very slow chart fetch:** Skeleton exceeds 10s → show timeout retry.
- **Mobile viewport:** Cards stack 1-col; charts full width; table horizontal scroll.

## Phase 2: Screen-by-Screen Functional Specifications

### Screen Name: Role Cockpit (`/pms/cockpit`)

* **Objective:** Present the correct KPI workbench for the authenticated user's role with live-ish data.
* **UI Elements & Field-Level Specs:**
  * **Header bar:**
    * Role `Badge` + dynamic title (e.g. "Department Manager Cockpit").
    * Last refreshed timestamp (`text-muted-foreground`).
    * `Button` refresh (`RefreshCw` icon) with loading spin when active.
    * `Select` auto-refresh interval.
  * **Summary cards row:** responsive grid `grid-cols-1 md:grid-cols-2 lg:grid-cols-4`.
    * Each `Card`: label, large value + unit, trend % with up/down icon, left border or dot in status color.
    * Cards clickable → drill-down ([3.1.6.2](./3.1.6.2-cockpit-drill-down.md)).
  * **Charts grid:** 2×2 on desktop; each `Card` with title + shadcn `Chart`.
    * Independent skeleton per chart.
    * Segments/bars/points clickable.
  * **Business table `Card`:**
    * `DataTable` with role-specific columns.
    * Row hover + `cursor-pointer`; keyboard Enter on focused row opens drill-down.
  * **Unsupported role:** centered `EmptyState` with `ShieldOff` icon.
* **Functional Logic:**
  * Mount: read `Role` from auth mock → fetch `RoleCockpitDto`.
  * Parallel fetch cards, charts, table (or single payload split client-side).
  * Auto-refresh timer re-fetches without remount; preserve expanded UI state.
  * Apply `getStatusColor(value, category)` to card accents.

### Screen Name: Widget Error State (inline on `/pms/cockpit`)

* **Objective:** Recover from single-widget failures without breaking the dashboard.
* **UI Elements & Field-Level Specs:**
  * `Alert` variant destructive compact inside widget slot: message + `Button` "Retry" size sm.
* **Functional Logic:**
  * Retry re-fetches only that widget's mock endpoint.

### Role → Cockpit Mapping (layout variants)

| Executive / CFO | Strategic cockpit | Group KPI cards, portfolio charts, alert summary |

| Auditor | Audit cockpit | Pending review counts, exception heatmap |

| 9 | Traffic rule change | [3.1.6.3](./3.1.6.3-traffic-light-rules.md) | Cockpit on next refresh | Colors update |

## Mock Data Shape

```ts
interface RoleCockpitDto {
  Role: string
  CockpitTitle: string
  LastRefreshedAt: string
  RefreshIntervalSec: number
  Cards: {
    Id: string
    Label: string
    Value: number
    Unit: string
    StatusColor: 'green' | 'yellow' | 'red'
    TrendPct: number | null
  }[]
  Charts: {
    Id: string
    Title: string
    Type: 'bar' | 'line' | 'donut'
    Series: { Label: string; Value: number }[]
  }[]
  Table: {
    Columns: string[]
    Rows: Record<string, string | number>[]
  }
}
```

Mocks: `src/mocks/pms/role-cockpit.ts` (one fixture per role).

## Acceptance Criteria

- [ ] Cockpit layout auto-selected from mock user role on load
- [ ] Data cards, charts, and table render with independent async states
- [ ] Manual refresh and configurable auto-refresh interval
- [ ] Per-widget error + retry without blocking entire page
- [ ] Traffic-light colors applied from global rule config
- [ ] Unsupported role shows empty state with CTA
- [ ] Clickable widgets hand off to drill-down interaction
- [ ] Four async states (loading / empty / error / success) at page and widget level
