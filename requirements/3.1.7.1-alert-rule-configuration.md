# 3.1.7.1 Global Alert Trigger Rule Custom Configuration — Frontend Requirements

**Source:** F-ALERT-001 — Section 3.1.7 Exception Alert & Management  
**Route:** `/pms/alerts/rules`, `/pms/alerts/rules/new`, `/pms/alerts/rules/:id/edit`  
**Related:** [3.7.5 BI alerts](./3.7.5-alerts-trend-forecasting.md) (analytics-oriented thresholds)  
**UI:** shadcn/ui — `DataTable`, `Switch`, `Badge`, `Button`, `Dialog`, `Form`, `Select`, `Input`, `Checkbox`, `Card`, `AlertDialog`, `Toast`, `Skeleton`, `DropdownMenu`

---

## Functional Summary

Managers configure exception alert trigger conditions: KPI overruns, project delays, filing overdues. Rules monitor business data 7×24 once enabled.

**Prerequisites:** Alert rule configuration and management permission.

---

## Phase 1: Deep Requirement Analysis

### User Personas

| Persona | Role | Goals | Pain points |
|---------|------|-------|-------------|
| **PMS / Ops Manager** | Rule author | Define thresholds and channels before incidents occur | Alert fatigue from poorly scoped rules |
| **Department Head** | Monitored object owner | Receive urgent alerts via right channel | SMS for low-priority items |
| **System Admin** | Governance | Bulk enable/disable during maintenance | No bulk operations |
| **Auditor** | Read-only | Review what conditions fire alerts | Rules opaque in natural language |

### Core Capabilities

| ID | Capability | Description |
|----|------------|-------------|
| FR-001 | Alert type taxonomy | KPI indicator, project progress, filing overdue |
| FR-002 | Condition builder | Field + operator + value composite trigger |
| FR-003 | Level & channel mapping | General / Important / Urgent with in-app, email, SMS multi-select |
| FR-004 | Rule CRUD | Create, edit, delete with enable toggle on list |
| FR-005 | Human-readable preview | Summary card before save plain-English rule |
| FR-006 | List filters | Type, level, enabled status |
| FR-007 | Bulk enable/disable | Multi-select rows + bulk action (mock) |
| FR-008 | Permission gate | Configuration role required |
| FR-009 | Last triggered column | Mock timestamp for operational awareness |

### Edge Cases & Alternative Paths

- **No channels selected:** Validation error "Select at least one channel."
- **Monitored object cleared after type change:** Reset picker when alert type changes.
- **Enable toggle on invalid rule:** Block with tooltip "Complete rule configuration first" (draft rules).
- **Bulk disable all:** Confirm dialog warning about monitoring gap.
- **Duplicate rule name:** Inline unique name validation mock.
- **Date vs numeric condition:** Condition value input switches type by field metadata.
- **Delete rule with open alerts:** Confirm "Existing alerts retained" message.

## Phase 2: Screen-by-Screen Functional Specifications

### Screen Name: Alert Rules List (`/pms/alerts/rules`)

* **Objective:** Manager maintains all alert trigger rules and their enabled state.
* **UI Elements & Field-Level Specs:**
  * Page header: "Alert Rules", `Button` "Create rule" primary.
  * Filter bar: `Select` type, `Select` level, `Select` enabled (all/yes/no), search by name `Input`.
  * Bulk bar (when rows selected): Enable, Disable `Button`s.
  * `DataTable` with row `Checkbox`: Rule name, Type `Badge`, Monitored object, Level `Badge` (semantic color), Channels (icons), Enabled `Switch`, Last triggered, Actions (`DropdownMenu`: Edit, Duplicate, Delete).
  * Level colors: General default, Important warning, Urgent destructive.
  * Channel icons: bell (in-app), mail, message-square (SMS) with `Tooltip`.
* **Functional Logic:**
  * Toggle `Switch` → immediate PATCH mock + toast.
  * Filters client-side or query mock API.
  * Delete → `AlertDialog` → remove from store.

### Screen Name: Rule Editor (`/pms/alerts/rules/new` or `/pms/alerts/rules/:id/edit`)

* **Objective:** Author complete alert rule with condition, level, channels, and preview.
* **UI Elements & Field-Level Specs:**
  * Breadcrumb: Alerts > Rules > Create / Edit.
  * Two-column layout: form left, preview right (stack on mobile).
  * **Form fields:**
    * Rule name — required `Input`.
    * Alert type — required `Select`: kpi | project | filing.
    * Monitored object — required searchable `Combobox` (options depend on type).
    * Trigger condition — row: Field `Select`, Operator `Select` (`>`, `<`, `=`, `>=`, `<=`), Value `Input` (number or date).
    * Alert level — required `Select`: general | important | urgent.
    * Push methods — `Checkbox` group: in-app, email, SMS (at least one).
    * Enabled on save — `Switch` default on.
  * **Preview `Card`:**
    * Human-readable sentence: "When {object} {field} {operator} {value}, send {level} alert via {channels}."
    * Sample `Badge` showing level color.
  * Footer: Cancel, Save primary.
* **Functional Logic:**
  * Changing type resets monitored object and condition field options.
  * Preview updates live from form watch.
  * Save → validate → POST/PATCH → toast → redirect to list.

### Screen Name: Channel Delivery Mock (dev panel optional)

* **Objective:** Demonstrate mocked delivery without real SMS/email.
* **UI Elements & Field-Level Specs:**
  * On save with channels: toast per channel "Mock: email queued" (console log panel in dev).
* **Functional Logic:**
