# 3.1.5.2 Manual Re-Calculation Triggered by Administrators — Frontend Requirements

**Source:** F-CALC-002 — Section 3.1.5 KPI Calculation and Statistical Analysis  
**Route:** `/pms/kpi/calculation/recalculate`, `/pms/kpi/calculation/recalculate/history`  
**UI:** shadcn/ui — `Card`, `Form`, `Select`, `Combobox`, `Button`, `Progress`, `Alert`, `Table`, `Badge`, `Banner`, `Toast`, `Skeleton`, `RadioGroup`

---

## Functional Summary

Admins manually trigger single-indicator or full-volume re-calculation after data corrections. Async execution without blocking UI; historical data retained.

**Prerequisites:** KPI calculation management permission.

---

## Phase 1: Deep Requirement Analysis

### User Personas

| Persona | Role | Goals | Pain points |
|---------|------|-------|-------------|
| **PMS Admin** | Calculation manager | Trigger targeted recalc after data fix without waiting for schedule | Full batch reruns waste time |
| **Data Steward** | Post-review corrector | Ensure corrected approved data reflects in KPI values quickly | Uncertainty whether recalc finished |
| **Auditor** | Compliance | Verify who triggered recalc and what was overwritten | No history of manual runs |

### Core Capabilities

| ID | Capability | Description |
|----|------------|-------------|
| FR-001 | Scope selection | All indicators, single indicator, or designated cycle/period |
| FR-002 | Async job execution | Start → queued → running → completed/failed without blocking navigation |
| FR-003 | Progress polling | Progress bar with `ProgressPct` mock polling every 2s |
| FR-004 | Non-blocking UI | Global banner when job in progress; user can navigate freely |
| FR-005 | Result overwrite display | Latest values replace display; history retained badge |
| FR-006 | Run history table | Timestamp, scope, operator, result count, status |
| FR-007 | Permission gate | Page hidden for unauthorized roles |
| FR-008 | Failure recovery | Failed job shows error `Alert` with retry action |

### Edge Cases & Alternative Paths

- **Second job started while one running:** Queue mock; toast "Job queued behind current run."
- **Scope = indicator but none selected:** Form validation blocks start.
- **Scope = cycle but invalid period:** Inline error on period picker.
- **Job fails mid-run:** Progress stops; `failed` status; partial `OverwrittenCount` shown.
- **User on page when job completes:** Toast success + results summary card animates in.
- **User navigated away:** Banner persists in app shell until job completes.
- **History empty:** Empty state on history tab.
- **Recalc with no approved data:** Job completes with zero overwrites; informational alert.

## Phase 2: Screen-by-Screen Functional Specifications

### Screen Name: Re-calculation Panel (`/pms/kpi/calculation/recalculate`)

* **Objective:** Admin configures scope and starts a manual re-calculation job.
* **UI Elements & Field-Level Specs:**
  * Page header: "Manual Re-calculation".
  * **Scope `Card`:**
    * `RadioGroup`: All indicators | Designated indicator | Designated cycle.
    * Conditional fields:
      * Indicator — `Combobox` (when designated indicator).
      * Cycle / period — `Select` + period `DatePicker` or label picker (when designated cycle).
    * `Button` "Start re-calculation" primary, disabled when job running.
  * **Active job `Card`** (visible when status queued/running):
    * Status `Badge`, `Progress` bar bound to `ProgressPct`.
    * Started by, started at, estimated remaining (mock).
    * `Button` "View history" ghost.
  * **Latest result `Card`** (visible when last job completed):
    * Overwritten count, completion time.
    * `Badge` "Previous run preserved — view history".
    * Link to history table.
  * Global in-progress `Alert` (also in app layout banner): "Re-calculation in progress…"
* **Functional Logic:**
  * Submit creates `RecalculationRequestDto`; starts mock polling.
  * On complete: stop poll, toast, refresh result card, append history.
  * On fail: destructive alert + retry button (creates new job).

### Screen Name: Re-calculation History (`/pms/kpi/calculation/recalculate/history`)

* **Objective:** Admin audits all manual re-calculation runs.
* **UI Elements & Field-Level Specs:**
  * `DataTable` columns: Run ID, Scope (label), Indicator/Cycle detail, Operator, Started at, Completed at, Status `Badge`, Overwritten count, History retained icon.
  * Filters: date range, scope type, status.
  * Row expand (optional): mock log snippet.
* **Functional Logic:**
  * Read-only; sort by Started at desc.
  * Click row → expand details or navigate to related job in [3.1.5.1](./3.1.5.1-scheduled-batch-calculation.md) if linked mock.

### Screen Name: Permission Denied (gate)

* **Objective:** Block unauthorized access gracefully.
* **UI Elements & Field-Level Specs:**
  * Empty state: lock icon, "Calculation management permission required", contact admin CTA.
* **Functional Logic:**
  * Route guard redirects or renders empty state.

## Phase 3: Screen Transition & User Flow Mapping

| # | Trigger | Source | Destination | Conditional paths |
|---|---------|--------|-------------|-------------------|
| 1 | Nav "KPI Calculation > Recalculate" | App shell | `/pms/kpi/calculation/recalculate` | No permission → denied state |
| 2 | Start re-calculation | Recalc panel | Same page (active job card) | Validation fail → stay |
| 3 | Job running | Any PMS route | Banner visible app-wide | — |
| 4 | Job completes | Poll | Same page (result card) | Fail → error alert |
| 5 | View history | Result card / link | `/pms/kpi/calculation/recalculate/history` | — |
| 6 | Data corrected in review | [3.1.4.3](./3.1.4.3-multi-level-data-review.md) | Recalc panel (admin flow) | Manual navigation |
| 7 | Retry failed job | Error alert | New queued job | — |
| 8 | Navigate away during job | Recalc panel | Any route | Banner persists until complete |
