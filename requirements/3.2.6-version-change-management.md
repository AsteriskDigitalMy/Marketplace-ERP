# 3.2.6 Version and Change Management — Frontend Requirements

**Source:** Section 3.2.6  
**Purpose:** Coordinated lifecycle version management for design, BOM, routing, and working hours — full traceability and rollback.

---

## Screens

### 1. Change request hub (`/pdm/changes`)

**Central inbox** for all PDM change applications.

| Column | |
|--------|--|
| Change # | Link — `CHG-2026-0089` |
| Entity type | Badge: Design / BOM / Routing / Working hours |
| Entity reference | Style # or name |
| Requested by | |
| Request date | |
| Reason (truncated) | Tooltip full text |
| Impact scope | Tags: Production, Procurement, Warehouse, Quality |
| Status | Draft / In review / Approved / Rejected / Applied |
| Version transition | v2 → v3 |

**Filters:** entity type, status, impact department, date range, requester, product.

**Actions:** `New change request`.

---

### 2. New change request (`/pdm/changes/new`)

**Stepped form:**

#### Step 1 — Select entity

| Field | Type |
|-------|------|
| Entity type | Select |
| Entity | Searchable picker (filtered by type) |
| Current version | Read-only |

#### Step 2 — Describe change

| Field | Required |
|-------|----------|
| Change reason | Textarea — min 20 chars |
| Change content | Textarea — what is changing |
| Impact scope | Multi-select: Production, Procurement, Warehouse, Quality, Finance, Sales |

#### Step 3 — Proposed changes

Entity-specific sub-form (mock diff preview):

| Entity type | UI |
|-------------|-----|
| Design | List document add/remove |
| BOM | Tree diff editor or link to BOM editor in change mode |
| Routing | Link to routing editor in change mode |
| Working hours | Hours field diff table |

#### Step 4 — Review & submit

Summary card + `Submit for approval`.

---

### 3. Change request detail (`/pdm/changes/:id`)

**Tabs:**

#### Overview

- All metadata, status banner, version transition
- Impact scope chips
- Approval timeline

#### Before / after comparison

- Side-by-side or unified diff view per entity type
- Highlight added (green), removed (red), modified (amber)

#### Notifications log (mock)

Table of departments notified on apply:

| Department | Notified at | Acknowledged by |
|------------|-------------|-----------------|
| Production | 2026-06-14 10:00 | — |
| Procurement | 2026-06-14 10:00 | Jane Doe |

#### Audit trail

Permanent archive entries — never deletable in UI.

---

### 4. Approval workflow

**Configurable steps (mock):** Requester manager → Technical → Affected department leads.

| Action | UI |
|--------|-----|
| Approve | Comment optional |
| Reject | Comment required |
| Request info | Status → `Pending info`; notifies requester (mock toast) |

---

### 5. Apply change (post-approval)

**On final approval (mock automation):**

1. Status → `Applied`
2. New version number auto-assigned (banner: "Now effective: v4")
3. Previous version preserved in history
4. **Notification dispatch panel** shows departments pushed (mock)
5. Rollback available via new change request only (not silent rollback)

**Apply confirmation `Dialog`:** restates impact scope before mock execution.

---

### 6. Version explorer (`/pdm/changes/versions`)

**Cross-entity version search.**

**Filters:**

- Product / style number
- Entity type
- Change type
- Date range
- Personnel (modifier or approver)

**Results table:**

| Product | Entity | Version | Effective date | Changed by | Change # | Actions |
|---------|--------|---------|----------------|------------|----------|---------|
| STL-1001 | BOM | v3 | 2026-06-01 | Jane | CHG-0082 | View, Compare |

**Compare (`Dialog`):** pick any two versions of same entity → diff view.

**Rollback UX:**

- No direct "rollback" button without change request
- `Create rollback change` CTA pre-fills reason: "Rollback to v{n}"

---

### 7. Global version indicators

Reuse across all PDM modules:

- **Header version badge** on entity detail pages
- **Pending change banner** when open change request exists: "Change CHG-0089 in review — editing locked"
- **Edit lock:** disable edit buttons when change in flight; tooltip explains why

## Mock Data Shape

```ts
type ChangeEntityType = 'design' | 'bom' | 'routing' | 'working_hours'
type ImpactDepartment = 'production' | 'procurement' | 'warehouse' | 'quality' | 'finance' | 'sales'

interface ChangeRequestDto {
  Id: string
  ChangeNumber: string
  EntityType: ChangeEntityType
  EntityId: string
  EntityReference: string   // style number or name
  VersionFrom: number
  VersionTo: number | null  // null until applied
  Reason: string
  ChangeContent: string
  ImpactScope: ImpactDepartment[]
  Status: 'draft' | 'in_review' | 'pending_info' | 'approved' | 'rejected' | 'applied'
  RequestedBy: { Id: string; DisplayName: string }
  RequestedDatetime: string
  AppliedDatetime: string | null
  ApprovalSteps: ApprovalStepDto[]
  Diff: ChangeDiffDto | null
  Notifications: { department: string; notifiedAt: string; acknowledgedBy: string | null }[]
}

interface ChangeDiffDto {
  Before: Record<string, unknown>
  After: Record<string, unknown>
  ChangedFields: string[]
}

interface VersionRecordDto {
  EntityType: ChangeEntityType
  EntityId: string
  ProductReference: string
  Version: number
  EffectiveDate: string
  ChangedBy: string
  ChangeRequestId: string
  ChangeNumber: string
}
```

Mocks: `src/mocks/pdm/changes.ts`.

## Cross-Module Integration (UI)

| Trigger | UI behavior |
|---------|-------------|
| Edit archived product | Redirect to new change request |
| BOM/routing/design save on archived entity | Block + CTA to change request |
| Change applied | Toast + update version badge on entity pages |

---

## Acceptance Criteria

- [ ] Change request hub lists all requests with filters
- [ ] Multi-step create form captures reason, content, impact, and entity diff
- [ ] Before/after comparison renders per entity type
- [ ] Approval workflow with approve, reject, and request-info paths
- [ ] Applied changes assign new version and show notification log
- [ ] Version explorer searchable by product, type, date, personnel
- [ ] Edit lock on entities with in-flight change requests
- [ ] Four async states on all views
