# 3.2.2.1 Product Project Initiation Management — Frontend Requirements

**Source:** Section 3.2.2.1  
**Workflow:** Project Initiation Application → Department Review → Technical Review → Management Approval → Project Takes Effect → Progress Tracking → Project Closure

---

## Screens

### 1. Project list (`/pdm/projects`)

**Layout:** Page title + filters + data table.

| Column | Sortable | Notes |
|--------|----------|-------|
| Project ID | Yes | Auto-generated after approval |
| Product name | Yes | Link to detail |
| Category | Yes | Badge |
| Customer | Yes | |
| Development lead | Yes | Avatar + name |
| Status | Yes | Workflow status chip |
| Development cycle | No | Date range |
| Budget | Yes | Currency formatted |
| Progress | No | Progress bar (% phases complete) |
| Overdue | No | Warning badge if past deadline |

**Filters:** status, category, customer, development lead, date range, overdue only.

**Actions:** `New project` (primary), export CSV (secondary).

**States:** loading skeleton table, empty with CTA, error with retry.

---

### 2. Project initiation form (`/pdm/projects/new`, `/pdm/projects/:id/edit`)

**Form:** React Hook Form + Zod.

#### Mandatory fields

| Field | Type | Validation |
|-------|------|------------|
| Product name | Text | Required, max 200 chars |
| Product category | Select | Required — Automotive Seat Leather, Aviation Interior Leather, Apparels, etc. |
| Customer name | Combobox | Required — searchable customer list (mocked) |
| Development cycle | Date range picker | Required, end ≥ start |
| Development lead | User picker | Required |
| Project budget | Currency input | Required, ≥ 0 |
| Development objectives | Textarea | Required, min 20 chars |

#### Optional / attachments

- File upload zone (shadcn dropzone pattern): design requirements, customer specs, reference drawings.
- Show file list with name, size, upload date, remove action.
- Accept common formats: PDF, DOCX, DWG, PNG, JPG (validate client-side).

#### Actions

- `Save draft` — saves without submitting approval.
- `Submit for approval` — opens confirmation dialog summarizing mandatory fields.

---

### 3. Project detail (`/pdm/projects/:id`)

**Tabs:**

1. **Overview** — all form fields read-only + status banner.
2. **Attachments** — file grid with preview/download.
3. **Approval** — workflow timeline (see below).
4. **Progress** — phase tracker (see below).
5. **Related** — links to quotations and sampling records (mocked).

**Header actions (contextual by status):**

| Status | Actions |
|--------|---------|
| Draft | Edit, Submit, Delete |
| In review | View only (+ approver actions if role permits) |
| Active | Update progress, Request closure |
| Closed | View only, Reopen (admin) |

---

### 4. Approval workflow UI

**Component:** `ApprovalTimeline` (shared).

**Steps (mocked workflow):**

1. Department review
2. Technical review
3. Management approval

Each step shows: approver name, status (`Pending` | `Approved` | `Rejected`), timestamp, comments.

**Approver view (mock role toggle in dev):**

- Approve / Reject buttons with required comment on reject.
- `Dialog` confirmation before submit.

On final approval: show success toast; project ID auto-displayed in header (`PRJ-2026-0042` format, mocked).

---

### 5. Progress tracking (`/pdm/projects/:id` → Progress tab)

**Phase list** (configurable phases, mocked):

| Phase | Status | Deadline | Owner |
|-------|--------|----------|-------|
| Initiation | Complete | — | — |
| Design | In progress | 2026-07-01 | Jane Doe |
| Sampling | Not started | 2026-08-15 | — |
| Finalization | Not started | — | — |

**Features:**

- Inline status update per phase (select: Not started, In progress, Complete, Blocked).
- Deadline date picker per phase.
- **Overdue alert:** if today > deadline and status ≠ Complete, show `Alert` (warning variant) on project list and detail header.
- Optional notification badge in sidebar nav (mocked count).

---

### 6. Project closure

**Dialog:** confirm closure with summary (completed phases, final notes textarea).

Post-closure: status → `Closed`; all edit actions disabled.

## Mock Data Shape

```ts
interface ProductProjectDto {
  Id: string
  ProjectCode: string | null          // null until approved
  ProductName: string
  ProductCategory: string
  CustomerName: string
  CustomerId: string
  DevelopmentCycleStart: string
  DevelopmentCycleEnd: string
  DevelopmentLead: { Id: string; DisplayName: string }
  ProjectBudget: number
  DevelopmentObjectives: string
  Status: 'draft' | 'department_review' | 'technical_review' | 'management_review' | 'active' | 'closed' | 'rejected'
  Progress: ProjectPhaseDto[]
  Attachments: AttachmentDto[]
  ApprovalSteps: ApprovalStepDto[]
  CreatedDatetime: string
  ModifiedDatetime: string | null
}

interface ProjectPhaseDto {
  PhaseName: string
  Status: 'not_started' | 'in_progress' | 'complete' | 'blocked'
  Deadline: string | null
  Owner: { Id: string; DisplayName: string } | null
  IsOverdue: boolean
}
```

Place mocks in `src/mocks/pdm/projects.ts`. Paginated list uses `PagedList<ProductProjectDto>`.
