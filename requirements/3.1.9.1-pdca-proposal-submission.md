# 3.1.9.1 Online Submission of Improvement Proposals by All Staff — Frontend Requirements

**Source:** F-PDCA-001 — Section 3.1.9 PDCA Improvement Closed-Loop Management  
**Routes:** `/pms/pdca/proposals`, `/pms/pdca/proposals/new`, `/pms/pdca/proposals/:id`  
**Related workflow:** Optional entry from [3.1.8.5](./3.1.8.5-hr-rectification.md) (HR rectification) → execution in [3.1.9.2](./3.1.9.2-pdca-execution-tracking.md)

---

## Functional Summary

All active employees submit online improvement proposals for production optimization, cost reduction, efficiency enhancement, and management optimization. Initial status is "Pending Evaluation"; auto-routed to Auditor review.

**Prerequisites:** Employee logged in.

---

## Phase 1: Deep Requirement Analysis

### User Personas

| Persona | Goals | Pain Points |
|---------|-------|-------------|
| **Employee (Submitter)** | Propose improvements; track evaluation status | No formal channel for ideas |
| **Auditor** | Receive structured proposals for evaluation | Incomplete problem descriptions |
| **HR Specialist** | Convert rectification into PDCA from [3.1.8.5](./3.1.8.5-hr-rectification.md) | Re-typing same content |

### Core Capabilities (FR-001…)

| ID | Capability | Priority |
|----|------------|----------|
| FR-001 | My proposals list with status badges | Must |
| FR-002 | New proposal form with category and required narrative fields | Must |
| FR-003 | Validation: min 50 chars on problem description | Must |
| FR-004 | Mock attachment upload (file names only) | Should |
| FR-005 | Submit → status `Pending evaluation`; mock Auditor inbox notification | Must |
| FR-006 | Pre-fill from HR rectification via `?fromHr=employeeId` [3.1.8.5](./3.1.8.5-hr-rectification.md) | Must |
| FR-007 | Proposal detail read-only view with status timeline | Must |
| FR-008 | Link to execution tracking when approved [3.1.9.2](./3.1.9.2-pdca-execution-tracking.md) | Should |

### Edge Cases & Alternative Paths

- **Pre-fill from HR — employee not found:** Empty form with warning banner.
- **Submit with validation errors:** Scroll to first invalid field.
- **Draft save (optional mock):** Not in original spec — skip unless mock supports local draft.
- **Rejected proposal:** Detail shows auditor comments; no re-submit in v1 (mock: create new).
- **Approved proposal:** Detail shows CTA to execution page [3.1.9.2](./3.1.9.2-pdca-execution-tracking.md).
- **Attachment over limit (mock):** Toast error max 5 files.
- **Unauthorized:** Redirect to login.

## Phase 2: Screen-by-Screen Functional Specifications

### Screen: My Proposals (`/pms/pdca/proposals`)

* **Objective:** Employee views all submitted improvement proposals and their statuses.
* **UI Elements & Field-Level Specs:**
  * Page title: "My Improvement Proposals"
  * Primary button: **New proposal** → `/pms/pdca/proposals/new`
  * Table columns: Proposal ID, Title, Category, Submitted at, Status (badge), Actions (**View**)
  * Status badges: Pending evaluation (yellow), Approved (green), Rejected (red), In execution (blue) — links to [3.1.9.2](./3.1.9.2-pdca-execution-tracking.md), Completed (gray)
  * Filter: Status, Category; search by title
  * Empty state: illustration + **Submit your first proposal**
  * Four async states
* **Functional Logic:**
  * List scoped to current user submitterId
  * View → `/pms/pdca/proposals/:id`

### Screen: Proposal Form — New (`/pms/pdca/proposals/new`)

* **Objective:** Capture a new improvement proposal for auditor evaluation.
* **UI Elements & Field-Level Specs:**
  * Breadcrumb: My Proposals → New proposal
  * Banner when `?fromHr=employeeId`: "Pre-filled from HR rectification" with link back to [3.1.8.5](./3.1.8.5-hr-rectification.md) workspace
  * Form fields:
    * Title (required, max 200 chars, counter)
    * Category (required select: Production / Cost / Efficiency / Management)
    * Current problem status (required textarea, min 50 chars, counter)
    * Improvement scheme (required textarea)
    * Expected results (required textarea)
    * Attachments (optional mock uploader — display file names, remove chip)
  * Footer: **Submit proposal** (primary), **Cancel** → list with unsaved confirm
* **Functional Logic:**
  * On `fromHr` load: map HR assistance summary → problem status; follow-up → improvement scheme; shortcomings → expected results; set `SourceHrRectificationId`
  * Submit: validate → mock API → toast → redirect to detail
  * Push mock banner to Auditor review inbox

### Screen: Proposal Detail — Submitter View (`/pms/pdca/proposals/:id`)

* **Objective:** Read-only view of submitted proposal with evaluation timeline.
* **UI Elements & Field-Level Specs:**
  * All form fields read-only
  * **Status timeline:** Submitted → Pending evaluation → Approved/Rejected → In execution (if applicable)
  * **Auditor comments** section (visible when evaluated)
  * CTA when Approved: **View execution** → `/pms/pdca/proposals/:id/execution` [3.1.9.2](./3.1.9.2-pdca-execution-tracking.md)
  * Source link if from HR: [3.1.8.5](./3.1.8.5-hr-rectification.md) employee context
  * **Back to list**
* **Functional Logic:**
  * 404 if not owner and not admin (mock: owner only for this spec)
  * Poll or refresh for status updates (mock manual refresh button)
