# 3.1.1.1 Organizational Unit Creation — Frontend Requirements

**Source:** F-BASE-001 — Section 3.1.1 System Basic Management  
**Route:** `/pms/admin/org`

---

## Phase 1: Deep Requirement Analysis

### User Personas

| Persona | Goals |
|---------|-------|
| **System Administrator (primary)** | Build and maintain the organizational hierarchy so downstream modules (KPI, projects, appraisals) can scope data correctly. Wants fast tree navigation, clear tier rules, and immediate feedback on validation errors. |
| **HR / Org Architect** | Mirror real factory structure (company → plant → line → shift) with correct process tags for KPI linkage. Needs confidence that sibling names and tier types are enforced. |
| **Auditor (read-only)** | Verify org structure without mutating it. Needs visible status badges and disabled-state indicators in the tree. |
| **Business Module Consumer (indirect)** | Relies on org pickers elsewhere in PMS. Expects only active units to appear after creation; does not interact with this screen directly. |

### Core Capabilities

1. **FR-001 — Org tree visualization:** Render a hierarchical, expandable tree of all organizational units with tier icons, sort order, and active/disabled status badges. Data sourced from `OrgUnit` in `src/models/pms/organization.ts`.
2. **FR-002 — Parent-scoped creation:** Allow creation of a child unit only under a selected active parent node via an **Add** action on the tree node context menu or toolbar.
3. **FR-003 — Tier-type enforcement:** Support all seven tier types (`company`, `plant`, `department`, `workshop`, `line`, `process`, `shift`) per `OrgTierTypeSchema`; constrain valid child tiers based on parent tier (mock rules: company → plant/dept; process requires `ProcessTag`).
4. **FR-004 — Process tag binding:** When `TierType = process`, require selection of a process tag from the global process dictionary (`DictionaryCategory` process entries from `src/models/pms/configuration.ts`).
5. **FR-005 — Sibling uniqueness validation:** Block submit when the new unit name duplicates an existing sibling under the same parent (client-side check against tree + server mock).
6. **FR-006 — Auto-generated org code:** On successful create, display system-generated `Code` in a **Toast**; append mock operation log entry.
7. **FR-007 — Async state handling:** Implement loading, empty, error, and success states for tree fetch and create submission.

### Edge Cases & Alternative Paths

- User selects a **disabled** parent → **Add** button disabled; tooltip: "Cannot add children under a disabled unit."
- User changes tier type from `process` to another after selecting a process tag → clear `ProcessTag` field and hide picker.
- User submits with network failure → **Toast** error; **Dialog** remains open with form values preserved.
- User double-clicks **Create** → button disabled after first click until request resolves.
- Tree has no nodes → empty state with CTA "Create root company" (mock: only one root company allowed).
- Duplicate sibling name detected on blur → inline field error before submit.
- User closes **Dialog** with unsaved changes → **Dialog** confirmation: "Discard unsaved changes?"
- Deep tree (10+ levels) → virtualized or lazy-load children on expand (mock: load all at once acceptable).

## Phase 2: Screen-by-Screen Functional Specifications

### Screen Name: Organization Tree (`/pms/admin/org`)

* **Objective:** Primary workspace for browsing the org hierarchy and initiating unit creation under a selected parent.

* **UI Elements & Field-Level Specs:**
  * **Page header:** Title "Organizational Structure"; subtitle with total unit count.
  * **Tree panel (shadcn `ScrollArea`):** Expand/collapse chevrons per node; node label shows `Name`, tier icon, `Code` (truncated), status **Badge** (`active` = default variant, `disabled` = secondary + strikethrough).
  * **Search input:** Text filter; debounced 300ms; highlights matching nodes and auto-expands ancestors.
  * **Toolbar — Add button (`Button`):** Enabled only when an active node is selected; `variant="default"`. Disabled state + tooltip when no selection or parent disabled.
  * **Toolbar — Refresh (`Button` `variant="ghost"`):** Reloads tree; shows skeleton during fetch.
  * **Empty state:** Illustration + "No organizational units yet" + primary **Button** to open create **Dialog** with `ParentId = null` (root company only).
  * **Loading state:** Tree skeleton (5 placeholder nodes).
  * **Error state:** Inline **Alert** with retry **Button**.

* **Functional Logic:**
  * On mount: `GET` mock org tree from `src/mocks/pms/org-structure.ts`; map to `OrgUnitDto[]`.
  * Node click: set `selectedParentId`; highlight row.
  * **Add** click: open **Create Organizational Unit Dialog** with `ParentId` pre-filled from selection.
  * On successful create from dialog: merge new node into tree at correct parent; expand parent; select new node; **Toast** success with generated `Code`.
  * Permission gate: hide **Add** if user lacks org management permission (mock).

### Screen Name: Create Organizational Unit Dialog (modal overlay on `/pms/admin/org`)

* **Objective:** Capture required fields for a new child unit and submit to mock create API.

* **UI Elements & Field-Level Specs:**
  * **Dialog (shadcn `Dialog`):** Title "Add Organizational Unit"; `max-w-lg`.
  * **Parent node (read-only `Input` or `Combobox` disabled):** Displays selected parent `Name` + tier; required; pre-filled; not editable.
  * **Organization name (`Input`):** Required; max 200 chars; placeholder "e.g. Cutting Workshop A"; validate on blur for sibling uniqueness.
  * **Tier type (`Select`):** Required; options filtered by parent tier (mock matrix); default empty; helper text per tier.
  * **Process business tag (`Combobox`):** Visible and required only when `TierType = process`; options from process dictionary; searchable; disabled otherwise.
  * **Sort order (`Input` type="number"):** Required; integer ≥ 0; default `0`; min validation.
  * **Footer — Cancel (`Button` `variant="outline"`):** Closes dialog; triggers unsaved-changes confirm if dirty.
  * **Footer — Create (`Button`):** `type="submit"`; disabled when form invalid or submitting; shows spinner when loading.

* **Functional Logic:**
  * Initialize form from `OrgUnitCreateRequestSchema` fields (`ParentId`, `Name`, `TierType`, `ProcessTag`, `SortOrder`).
  * Real-time: show/hide process tag field on tier change; clear tag when tier ≠ process.
  * Submit: validate with Zod (`OrgUnitSchema` superRefine for process tag); `POST` mock; prevent double-submit.
  * Success: close **Dialog**; parent screen **Toast** `"Unit created — Code: {Code}"`; mock operation log write.
  * Error — duplicate name: inline error on name field; remain open.
  * Error — server: **Toast** destructive variant with message; re-enable submit.

## Phase 3: Screen Transition & User Flow Mapping

| # | Trigger Event | Source Screen | Destination Screen | Conditional Paths |
|---|---------------|---------------|--------------------|-------------------|
| 1 | Page load / deep link | — | Organization Tree (`/pms/admin/org`) | Permission denied → mock 403 page |
| 2 | Click tree node | Organization Tree | Organization Tree (same) | Updates selection highlight only |
| 3 | Click **Add** with active parent selected | Organization Tree | Create Organizational Unit Dialog | No selection → button disabled, no transition |
| 4 | Click **Create root company** (empty state) | Organization Tree | Create Organizational Unit Dialog | Only when tree empty |
| 5 | Click **Create** (valid form) | Create Dialog | Organization Tree | Success → dialog closes, tree updates |
| 6 | Click **Cancel** or overlay close (no changes) | Create Dialog | Organization Tree | — |
| 7 | Click **Cancel** with dirty form | Create Dialog | Confirm discard Dialog → Organization Tree | User chooses "Stay" → remain in Create Dialog |
| 8 | Create API failure | Create Dialog | Create Dialog | Toast error; form preserved |
| 9 | Click **Refresh** | Organization Tree | Organization Tree | Shows loading skeleton during refetch |
