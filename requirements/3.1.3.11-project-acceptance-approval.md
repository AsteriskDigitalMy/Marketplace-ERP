# 3.1.3.11 Project Completion Acceptance Approval — Frontend Requirements

**Source:** F-PROJ-011 — Section 3.1.3 Project Management  
**Route:** `/pms/projects/acceptance-reviews`  
**Related:** [3.1.3.10](./3.1.3.10-project-acceptance-initiation.md), [3.1.3.8](./3.1.3.8-kpi-data-sync.md), [3.1.3.1](./3.1.3.1-project-initiation.md), [3.1.3.2](./3.1.3.2-project-initiation-approval.md) (similar workflow pattern)

---

## Functional Summary

Auditors verify overall project outcomes. Pass → archived/sealed permanently; fail → return with rectification requirements.

**Prerequisites:** Final project acceptance permission; status `Pending acceptance`.

---

## Phase 1: Deep Requirement Analysis

### User Personas

| Persona | Role | Goals | Pain Points |
|---------|------|-------|-------------|
| **Acceptance Auditor** | Final gatekeeper | Objective evaluation with evidence | Large attachment packages |
| **Project Leader** | Awaits seal | Clear pass/fail criteria | Vague rectification requirements on fail |
| **Archive Researcher** | Later business review | Find sealed projects read-only | Archived projects hard to discover |

### Core Capabilities (FR-xxx)

| ID | Capability | Priority |
|----|------------|----------|
| FR-001 | Acceptance review inbox at `/pms/projects/acceptance-reviews` | Must |
| FR-002 | Review detail tabs: Execution materials, Task summary, Attachments, KPI achievement | Must |
| FR-003 | Decision form: comprehensive opinion, grade A/B/C/D, Pass/Fail | Must |
| FR-004 | Pass → status `archived`; read-only seal `Badge` | Must |
| FR-005 | Fail → rectification requirements + deadline; return to leader [3.1.3.10](./3.1.3.10-project-acceptance-initiation.md) | Must |
| FR-006 | Archived projects searchable in archive list (mock) | Should |
| FR-007 | KPI achievement tab uses [3.1.3.8](./3.1.3.8-kpi-data-sync.md) data | Must |

### Edge Cases & Alternative Paths

- **Empty inbox:** Empty state illustration.
- **Fail without rectification text:** Validation block.
- **Pass on partial target achievement:** Warning confirm if application said `not_met`.
- **Already reviewed:** Stale conflict toast.
- **Archived project edit attempt:** All actions disabled; seal tooltip.
- **Attachment preview fail:** Placeholder with download mock link.

## Phase 2: Screen-by-Screen Functional Specifications

### Screen: Acceptance Review Inbox — `/pms/projects/acceptance-reviews`

* **Objective:** List projects awaiting final acceptance decision.

* **UI Elements & Field-Level Specs:**
  - `Card` title "Project Acceptance Reviews" with pending `Badge` count.
  - `DataTable`: Project Code, Name, Leader, Target Achievement (`Badge` from application), Submitted At, Actions (`Button` "Review").
  - `Input` search; sort by submitted date desc.
  - Link to **Archived Projects** mock list filter on `/pms/projects?status=archived`.

* **Functional Logic:**
  - Filter projects `Status === pending_acceptance`.
  - Review → `/pms/projects/acceptance-reviews/:projectId`.

---

### Screen: Review Detail — `/pms/projects/acceptance-reviews/:projectId`

* **Objective:** Comprehensive review and pass/fail decision.

* **UI Elements & Field-Level Specs:**
  - `Breadcrumb`: Acceptance Reviews → `{Project Code}`.
  - `Tabs`:
    - **Execution materials** — read-only initiation + execution summary from [3.1.3.10](./3.1.3.10-project-acceptance-initiation.md).
    - **Task summary** — `DataTable` all tasks: name, owner, final progress, acceptance criteria met `Badge`.
    - **Outcome attachments** — file list with mock preview/download `Button`s.
    - **KPI achievement** — table from [3.1.3.8](./3.1.3.8-kpi-data-sync.md): KPI, synced value, target, variance `Badge`.
  - **Decision form** (sticky footer `Card`):
    - **Comprehensive opinion** — `Textarea`, required.
    - **Overall evaluation grade** — `Select`: A | B | C | D.
    - **Decision** — `RadioGroup`: Pass | Fail.
    - **Rectification requirements** — `Textarea`, required when Fail; **Deadline** — `Calendar` when Fail.
  - Actions: Back | Submit Decision.

* **Functional Logic:**
  1. Pass: status `archived`; seal `Badge` "Archived — Sealed"; project read-only everywhere.
  2. Fail: store rectification fields; status returns to `in_progress` or `acceptance_returned` mock; notify leader; link to [3.1.3.10](./3.1.3.10-project-acceptance-initiation.md) resubmit.
  3. Archived projects appear in archive browse list.

---

### Screen: Archived Project View — `/pms/projects/:id` (read-only)

* **Objective:** Browsable sealed project for later business review.

* **UI Elements & Field-Level Specs:**
  - Prominent seal `Badge` with `Lock` icon at header.
  - All tabs read-only; no edit/create `Button`s.
  - Banner: "This project is permanently archived".

* **Functional Logic:**
  - Any edit action hidden or disabled with `Tooltip`.

---

### Modal: Confirm Pass — `AlertDialog`

* **Objective:** Confirm permanent archive implications.

* **UI Elements & Field-Level Specs:** Warning text about irreversible seal; Confirm / Cancel.

* **Functional Logic:** Shown on Pass submit (optional always-on for compliance).

## Phase 3: Screen Transition & User Flow Mapping

| # | Trigger | Source | Destination | Conditional Paths |
|---|---------|--------|-------------|-------------------|
| 1 | Nav "Acceptance Reviews" | App shell | `/pms/projects/acceptance-reviews` | Auditor permission |
| 2 | Click Review | Inbox row | `/pms/projects/acceptance-reviews/:projectId` | — |
| 3 | Submit Pass | Review detail | Inbox | → project `archived` + seal |
| 4 | Submit Fail | Review detail | Inbox | → leader [3.1.3.10](./3.1.3.10-project-acceptance-initiation.md) |
| 5 | Leader fail notification | Toast mock | `/pms/projects/:id/acceptance` | Resubmit flow |
| 6 | KPI tab link | Review detail | `/pms/projects/:id/kpi-sync` | [3.1.3.8](./3.1.3.8-kpi-data-sync.md) |
| 7 | Archive browse | Inbox link | `/pms/projects?status=archived` | Read-only list |
| 8 | Open archived project | Archive list | `/pms/projects/:id` | Seal badge; no edits |

## Acceptance Criteria

- [ ] Acceptance review inbox `DataTable` with search and sort
- [ ] Review detail `Tabs` for materials, tasks, attachments, KPI achievement ([3.1.3.8](./3.1.3.8-kpi-data-sync.md))
- [ ] Decision form with opinion, grade `Select`, Pass/Fail `RadioGroup`
- [ ] Fail requires rectification requirements and deadline
- [ ] Pass archives project with permanent seal `Badge` and read-only UI
- [ ] Archived projects browsable via filtered project list
- [ ] Pass confirmation `AlertDialog` for archive permanence
- [ ] Leader notification and resubmit path to [3.1.3.10](./3.1.3.10-project-acceptance-initiation.md)
- [ ] Four async states: loading, empty inbox, stale review error, success redirect
