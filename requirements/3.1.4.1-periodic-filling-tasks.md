# 3.1.4.1 Automatic Push of Periodic Data-Filling Tasks — Frontend Requirements

**Source:** F-DATA-001 — Section 3.1.4 Data Collection and Filling  
**Route:** `/pms/data-collection/tasks` (admin), `/pms/data-collection/my-tasks` (assignee), `/pms/data-collection/rules` (admin)  
**UI:** shadcn/ui — `Card`, `Table`, `DataTable`, `Badge`, `Button`, `Dialog`, `AlertDialog`, `Form`, `Select`, `Input`, `DatePicker`, `Switch`, `Toast`, `Skeleton`, `EmptyState`

---

## Functional Summary

System issues periodic filling tasks per indicator rules and positions. Dual reminder (pop-up + message). Overdue triggers alert and log.

**Prerequisites:** KPI set to manual filling; preset filling time reached (mock scheduler).

---

## Phase 1: Deep Requirement Analysis

### User Personas

| Persona | Role | Goals | Pain points |
|---------|------|-------|-------------|
| **PMS Admin** | KPI / data-collection administrator | Configure filling cycles, assign positions, monitor overdue tasks | Manual spreadsheet tracking; missed deadlines without visibility |
| **Data Filler** | Shop-floor or department employee | See due tasks quickly, dismiss reminders, navigate to fill form | Login without knowing what is due; reminder fatigue |
| **Department Manager** | Supervisor (read-only on rules) | Glance at team overdue counts | No aggregate view of team compliance |

### Core Capabilities

| ID | Capability | Description |
|----|------------|-------------|
| FR-001 | Filling rule CRUD | Admin creates/edits rules: indicator, cycle, target positions, template binding, due offset |
| FR-002 | Task auto-generation (mock) | On cycle trigger, system generates `DataFillingTaskDto` rows per assignee |
| FR-003 | Personal to-do list | Assignee sees tasks with due date, countdown badge, and status |
| FR-004 | Dual reminder | On login: in-station `Dialog` pop-up + persistent in-app message badge for due tasks |
| FR-005 | Overdue handling | Overdue tasks show destructive `Badge`; mock alert record created (links to [3.1.7.1](./3.1.7.1-alert-rule-configuration.md)) |
| FR-006 | Session reminder throttle | Pop-up shown once per session/day per user (mock `sessionStorage` flag) |
| FR-007 | Async state coverage | Loading, empty, error, success on list, rules, and pop-up data fetches |

### Edge Cases & Alternative Paths

- **No manual-fill indicators configured:** Rules list empty state with CTA to KPI library ([3.1.2.1](./3.1.2.1-kpi-indicator-addition.md)).
- **Assignee has zero tasks:** My-tasks empty state — "No filling tasks assigned."
- **All tasks submitted:** Pop-up skipped; success empty variant on my-tasks.
- **Pop-up dismissed but tasks remain:** Message center badge persists until all due tasks submitted or overdue resolved.
- **Rule edited mid-cycle:** Existing pending tasks unchanged; new cycle uses updated rule (mock note in toast).
- **Position unbound from user:** Task not generated for that user; admin sees warning on rule save if no assignees match.
- **Scheduler mock failure:** Rules list loads; task generation shows inline error with retry.
- **Overdue → filler submits late:** Status transitions `overdue` → `submitted`; alert record remains with "resolved" mock flag.

## Phase 2: Screen-by-Screen Functional Specifications

### Screen Name: Filling Rules List (`/pms/data-collection/rules`)

* **Objective:** Admin views and manages all periodic filling rules that drive task generation.
* **UI Elements & Field-Level Specs:**
  * Page header: title "Filling Rules", primary `Button` "Add rule" (`Plus` icon).
  * Filter bar: `Select` indicator (searchable), `Select` cycle (daily/weekly/monthly/quarterly), `Input` search by position name.
  * `DataTable` columns: Indicator name, Cycle, Target positions (truncated + tooltip), Template name, Due offset (days), Active toggle (`Switch`), Last triggered, Actions (`DropdownMenu`: Edit, Disable, Delete).
  * Row active `Switch` toggles rule without opening editor; confirm `AlertDialog` on disable.
  * Pagination: 20 rows default; server-side mock.
* **Functional Logic:**
  * Fetch rules on mount; filter client-side or mock API query params.
  * Delete requires confirmation; disabled rules do not generate new tasks.
  * Click Edit → Rule Editor dialog. Click Add → empty Rule Editor.

### Screen Name: Filling Rule Editor (Dialog on `/pms/data-collection/rules`)

* **Objective:** Admin configures one periodic filling rule end-to-end.
* **UI Elements & Field-Level Specs:**
  * `Dialog` (lg width): `Form` with fields:
    * Indicator — required `Combobox` from KPI library (manual-fill only).
    * Cycle — required `Select`: daily, weekly, monthly, quarterly, annual.
    * Target positions — required multi `Combobox` from org positions.
    * Template — required `Select` from form templates mock.
    * Due offset — required `Input` type number (days after period start).
    * Active — `Switch`, default on.
  * Footer: Cancel, Save (`Button` primary).
  * Inline `FormMessage` for validation errors.
* **Functional Logic:**
  * Validate all required fields before save.
  * Save → toast success → close dialog → refresh list.
  * Unsaved changes → `AlertDialog` on close attempt.

### Screen Name: Admin Task Monitor (`/pms/data-collection/tasks`)

* **Objective:** Admin monitors all generated filling tasks across the organization.
* **UI Elements & Field-Level Specs:**
  * Summary `Card` row: Pending count, Submitted count, Overdue count (traffic-light badges).
  * Filters: period, status, assignee, indicator.
  * `Table`: Task name (indicator + period), Assignee, Due date, Status `Badge`, Actions (View — read-only link to fill form if submitted).
  * Overdue rows: `Badge` variant destructive + row highlight `bg-destructive/5`.
* **Functional Logic:**
  * Sort by due date ascending default.
  * Click overdue row → optional link to alert record mock.

### Screen Name: Employee To-Do (`/pms/data-collection/my-tasks`)

* **Objective:** Assignee sees personal filling tasks and navigates to entry form.
* **UI Elements & Field-Level Specs:**
  * Page title "My Filling Tasks".
  * `Table` or `Card` list: Task name, Due date (with countdown `Badge`: "3 days left" / "Due today" / "Overdue"), Status, Action `Button` "Fill" or "View".
  * Status badges: Pending (`secondary`), Submitted (`default`), Overdue (`destructive`).
  * Empty state illustration + "No tasks assigned."
* **Functional Logic:**
  * "Fill" navigates to `/pms/data-collection/fill/:taskId` ([3.1.4.2](./3.1.4.2-manual-data-entry.md)).
  * Submitted tasks: "View" opens read-only form.

### Screen Name: Login Reminder Pop-up (Global overlay)

* **Objective:** Surface due filling tasks immediately after authentication.
* **UI Elements & Field-Level Specs:**
  * `Dialog` modal, non-dismissible backdrop optional; title "Filling tasks due".
  * `Table` or list: up to 5 due tasks with indicator, period, due date.
  * Footer: "Remind me later" (`Button` outline), "Go to my tasks" (`Button` primary).
  * `Checkbox` "Don't show again today" (sets session flag).
* **Functional Logic:**
  * Trigger on app load when mock auth resolves and due `pending` tasks exist.
  * Respect once-per-day/session flag in `sessionStorage`.
  * Parallel: increment message center badge count for due tasks.

## Phase 3: Screen Transition & User Flow Mapping

| # | Trigger | Source | Destination | Conditional paths |
|---|---------|--------|-------------|-------------------|
| 1 | Nav "Data Collection > Rules" | App shell | `/pms/data-collection/rules` | Permission gate → 403 empty |
| 2 | Click "Add rule" | Rules list | Rule Editor dialog | — |
| 3 | Save rule | Rule Editor | Rules list (refresh) | Validation fail → stay in dialog |
| 4 | Nav "My Tasks" | App shell | `/pms/data-collection/my-tasks` | — |
| 5 | Click "Fill" | My tasks row | `/pms/data-collection/fill/:taskId` | Task already submitted → read-only |
| 6 | Login success | Auth | Reminder pop-up (overlay) | No due tasks → skip pop-up |
| 7 | "Go to my tasks" | Pop-up | `/pms/data-collection/my-tasks` | Close pop-up |
| 8 | "Remind me later" | Pop-up | Current page | Set session dismiss flag |
| 9 | Task becomes overdue (mock) | Scheduler tick | My tasks + Admin tasks | Badge destructive; alert record created |
| 10 | Nav "Task Monitor" | App shell | `/pms/data-collection/tasks` | Admin role only |
